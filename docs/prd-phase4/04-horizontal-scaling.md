# 04. Horizontal Scaling — 워커 N 함수 형태 · 수평 확장 SLO · Redis knee point

> 담당 페르소나: **Performance Engineer** · Architect 보조 (Redis 한계 식별)
> 본 문서는 4단계 PRD 의 **수평 확장 SLO + Redis 단일 인스턴스 knee point** 단일
> 출처를 잠근다. 1~2단계가 잠근 `SERVICE_MODE=worker` + `docker compose up --scale
> worker=N` 의 함수 형태(처리량(N) / p99(N) / 큐 길이(N)) 를 측정하고, Redis 단일
> 인스턴스가 어느 부근에서 한계에 도달하는지 식별한다.
> 구현 코드(워커 N 매트릭스 측정 자동화 + knee point 식별 절차) 는 본 PRD 가
> 승인된 뒤 **후속 PLAN 단계** 에서 작성한다.

## 1. 컨텍스트 / 배경

본 시스템은 1~2단계에서 워커의 수평 확장 기반을 잠근다.

- `SERVICE_MODE=worker` 환경변수로 워커 전용 프로세스 분기 (`packages/demo/src/server.ts`).
- `docker compose up --scale worker=N` 으로 N 인스턴스 기동 (docker-compose.yml).
- 워커들은 같은 Redis 큐를 공유 소비(BullMQ 의 표준 동작 — 한 작업은 한 워커에만
  잡힌다).
- `architecture.md` §2 가 본 구조를 잠금.

그러나 **N 이 늘어날 때 처리량 / p99 / 큐 길이가 어떻게 움직이는지** 는 측정한 적이
없다(`00-overview` §1 정합). 본 PRD `04` 는 이 공백을 채운다.

- 워커 N 의 함수 형태를 측정해 본 시스템의 **수평 확장 SLO** 를 정의한다(§3 / §4).
- 처리량(N) 의 선형성 / 포화 / 역전(degradation) 패턴을 식별한다.
- Redis 단일 인스턴스의 **knee point** (CPU / 메모리 / 네트워크 한계 도달 지점) 를
  식별해 HA / Cluster 도입의 트리거 조건을 명문화한다(§5 / §6). Q-LOAD-4 (a) 결정
  정합 — 도입 자체는 본 PRD 범위 밖.

> 본 문서가 잠그는 것은 **수평 확장 SLO 의 형태와 knee point 식별 절차** 이지,
> 정확한 선형성 허용 오차 α / β 의 숫자가 아니다. 후자는 Q-LOAD-10 / Q-LOAD-11
> 결정 위임 — §4 가 옵션 정리 + 결정 라인을 명시한다.

## 2. 수평 확장의 정의

### 2.1 측정 변수 N (워커 수)

본 PRD 가 측정하는 워커 수 N 의 매트릭스:

| N | docker-compose 명령 | 의도 |
|---|---------------------|------|
| 1 | `docker compose up` (기본) | 기준선. 단일 워커 처리량 / p99 / 큐 길이 |
| 2 | `docker compose up --scale worker=2` | 선형성의 첫 단계 확인 |
| 5 | `docker compose up --scale worker=5` | 중간 영역 — knee point 후보 |
| 10 | `docker compose up --scale worker=10` | 포화 / 역전 식별 영역 |

**총 4 값 × 4 LP-ID = 16 측정.** 본 PRD `01` 의 LP-N 카탈로그와 N 매트릭스가
직교(`01` §2.5 정합 — LP-N 은 N 에 직교).

LP-1~LP-4 의 W (측정 윈도우, `01` §5) 와 결합 시 총 측정 시간:
- LP-1 / LP-4 (짧은 측정, 약 7분) × 4 N = 약 28분 × 2 LP = 약 56분.
- LP-2 / LP-3 (sustained 측정, 약 32분) × 4 N = 약 128분 × 2 LP = 약 256분.
- **합계: 약 5.2시간** (워밍업 / 쿨다운 / docker compose down + up 의 셋업 시간 별도).

### 2.2 N 매트릭스의 잠정 폐쇄성

- 본 PRD 는 N ∈ {1, 2, 5, 10} 만 측정한다. N = 3 / 4 / 7 / 20 등 추가 측정은
  본 PRD 본문 갱신 PR 을 요구한다.
- 본 PRD 의 4 값은 knee point 식별의 최소 셋이다. 더 세밀한 식별이 필요해지면
  PLAN 단계가 (예: N = 7, 8, 9) 미세 매트릭스를 추가 측정.
- N = 0 (워커 없음) 은 정의상 의미 없으므로 매트릭스 외.

### 2.3 측정 환경 (Q-LOAD-2 (b) 정합)

본 PRD `02` §4.2 의 cgroup 한정값 표가 단일 출처. 워커 수 N 에 따라 다음을 확인한다.

- N × worker cgroup `cpus` (예: 5 × 1.0 = 5 코어) ≤ 측정 호스트 코어 수 - α (다른
  서비스용 여유).
- N 가 호스트 사양을 초과하면 cgroup 격리 의미가 약해진다 — `02` §4.2 메모.
- 호스트 사양 한계 시: 측정 결과 메타데이터에 over-commit 여부 기록 (결과 해석에
  영향).

## 3. 처리량 / 지연 / 큐 길이의 함수 형태

본 §3 은 본 PRD 가 측정하는 N 의 함수 형태 단일 출처다.

### 3.1 이론적 함수 형태

| 함수 | 이론적 패턴 | 본 PRD 의 측정 의도 |
|------|---------------|----------------------|
| 처리량(N) = `사용자가 본 시스템이 처리하는 RPS` | 선형 (ideal): `처리량(N) = 처리량(1) × N` | 선형성에서 얼마나 벗어나는지 |
| p99(N) = `워커 처리 지연 p99` | 상수 (ideal): `p99(N) ≈ p99(1)` | N 증가 시에도 p99 가 안정한지 |
| 큐 길이(N) = `정상 부하 하 큐 적재량` | 0 근방 (ideal): 워커가 즉시 소비 | N 증가 시 큐 적재가 줄어드는지 |
| Redis CPU(N) | 선형 증가 → 포화 (이론적 knee) | Redis 단일 인스턴스의 한계 식별 |

### 3.3 선형성 평가 형식

PLAN 측정 결과의 선형성 평가:

```
선형성(N) = 처리량(N) / (처리량(1) × N)
         = (실측 처리량 N 워커) / (이상 처리량 N 워커)

선형성(N) = 1.0  → 완전 선형 (ideal)
선형성(N) < 1.0  → 선형성 손실 (자원 경합 / 락 / Redis 포화)
선형성(N) > 1.0  → 측정 노이즈 또는 1 워커 측정의 비효율 (드물게 가능)
```

본 PRD 의 수평 확장 SLO (§4) 가 선형성의 허용 하한 α 를 잠근다.

## 4. 수평 확장 SLO 형태 — Q-LOAD-10 / Q-LOAD-11 결정 위임

본 §4 는 본 PRD 가 새로 정의하는 수평 확장 SLO 의 형태 단일 출처. 3단계 SLO-1~4
가 절대 임계로 정의된 것과 달리, 본 SLO 는 **N 의 함수로 정의된 상대 임계** 다.

> **결정 위임 안내:** Q-LOAD-10 (선형성 허용 오차 α) 와 Q-LOAD-11 (수평 확장 SLO 의
> 정확한 형태) 은 본 문서 §4 가 옵션 정리 + 결정 라인 명시. 본 PRD `00-decisions-needed.md`
> §🟡 표의 Q-LOAD-10 위치(`04-horizontal-scaling.md` §3) 표기는 본 §4 와 동일
> 가리킴 — 본 PRD 본문 §4 가 단일 출처.

### 4.1 처리량 선형성 SLO 형태

본 PRD 가 잠그는 형태:

```
처리량(N) ≥ 처리량(1) × N × α
```

`α` 의 옵션:

| 옵션 | α | 의미 |
|------|---|------|
| (a) 보수 | 0.7 | 30% 손실 허용. 단일 호스트 + cgroup 환경의 자원 경합 흡수 |
| (b) 중도 | 0.8 | 20% 손실 허용. Google SRE 일반 권고 영역 |
| (c) 공격 | 0.9 | 10% 손실 허용. 거의 완전 선형성 요구 |

**트레이드오프:**
- (a) α=0.7 — 측정 환경 노이즈 흡수. 본 시스템의 수평 확장 능력을 약하게만 약속.
- (b) α=0.8 — 본 PRD 가 가장 합리적이라 판단하는 잠정 기본값. 단일 호스트 환경의
  자원 경합을 부분 흡수하면서도 수평 확장 능력을 의미 있게 약속.
- (c) α=0.9 — 거의 완전 선형성. 본 PRD 측정 환경(단일 호스트 + cgroup) 에서 도달
  어려울 수 있음. 단독 개발 + 데모 패턴에는 과잉.

**잠정 권고:** (b) α=0.8. PLAN 측정 결과가 (b) 를 만족하지 못하면 측정 환경 / 본
시스템의 자원 경합 패턴 재검토 필요. PLAN 진입 전 사용자 잠금.

### 4.2 p99 안정성 SLO 형태

본 PRD 가 잠그는 형태:

```
p99(N) ≤ p99(1) × β
```

`β` 의 옵션:

| 옵션 | β | 의미 |
|------|---|------|
| (a) 엄격 | 1.1 | N 증가 시 p99 가 10% 이내 증가 허용. 매우 엄격 |
| (b) 중도 | 1.2 | 20% 이내 증가 허용. 일반 권고 |
| (c) 느슨 | 1.5 | 50% 이내 증가 허용. 큐 경합 / Redis 응답 지연 흡수 |

**트레이드오프:**
- (a) β=1.1 — 본 시스템의 p99 가 N 의 함수로 거의 안정함을 약속. 단독 호스트 +
  cgroup 환경에서 도달 어려울 수 있음.
- (b) β=1.2 — 본 PRD 가 가장 합리적이라 판단하는 잠정 기본값. Q-LOAD-10 (a) α=0.7
  과 결합 시 보수적 SLO. Q-LOAD-10 (b) α=0.8 과 결합 시 중도 SLO.
- (c) β=1.5 — 큐 경합 / Redis 응답 지연을 의도적으로 흡수. p99 안정성을 약하게 약속.

**잠정 권고:** (b) β=1.2. PLAN 진입 전 사용자 잠금.

### 4.3 큐 길이 회복 시간 SLO (LP-4 spike 전용)

LP-4 (spike) 측정에서 `R_spike` 인가 후 큐 길이가 baseline 으로 회복되는 시간:

```
회복 시간(N) ≤ 회복 시간(1) × γ
```

`γ` 는 본 PRD 가 잠금 시도하지 않는다 — Q-LOAD-11 의 결정 자체가 (b) p99 안정성을
포함하는 형태일 경우 LP-4 회복 시간은 본 SLO 의 부속 측정값으로 간주. Q-LOAD-11
결정 후 본 §4.3 의 정식 잠금 여부를 PLAN 단계가 추가 검토.

### 4.4 수평 확장 SLO 카탈로그 (Q-LOAD-11 결정 후 잠금)

본 §4.4 는 Q-LOAD-11 결정 후 잠금될 표의 형식 단일 출처.

| SLO ID | 한 줄 | SLI | 목표 (α / β) |
|--------|-------|-----|----------------|
| SLO-H-1 (수평 확장 처리량) | N 의 함수로 처리량이 선형 영역 안에 있다 | `sum(rate(webhook_relay_jobs_processed_total{job_state="completed"}[5m]))` | `처리량(N) ≥ 처리량(1) × N × α`, α = (Q-LOAD-10) |
| SLO-H-2 (수평 확장 p99 안정성) | N 증가 시 p99 처리 지연이 안정 | `histogram_quantile(0.99, sum by (le) (rate(webhook_relay_worker_processing_duration_seconds_bucket{outcome="success"}[5m])))` | `p99(N) ≤ p99(1) × β`, β = (Q-LOAD-11) |

본 §4.4 의 SLI PromQL 은 3단계 메트릭 카탈로그(`prd-phase3/01` §3) 의 메트릭을 그대로
사용 — 새 메트릭 도입 0건 (`00-overview` §2 G4.3 / §3 N4.6 정합).

### 4.5 수평 확장 SLO 의 절대 임계와의 정합

- **3단계 SLO 와의 관계:** SLO-H-1 / SLO-H-2 는 N 의 함수로 정의된 **상대** SLO 다.
  3단계 SLO-1~4 는 **절대** SLO. 두 SLO 체계는 보완적이며 동시에 만족해야 한다.
- **예시:** N=10 측정 결과가 SLO-H-2 의 `p99(10) ≤ p99(1) × 1.2` 를 만족해도, 3단계
  SLO-3 의 `p99 ≤ 5s` 절대 임계를 위반하면 시스템 SLO 위반. 둘 다 만족해야 정상.
- **알람 발화:** 3단계 알람 규칙(`prd-phase3/04` §5.2) 은 절대 임계만 다룬다. SLO-H-1
  / SLO-H-2 의 위반 감지는 사람 검토 + 결과 보고서 commit 으로. 본 PRD 가 새 알람을
  도입하지 않는다 (`00-overview` §3 N4.1 / N4.6 정합).

## 5. Redis 단일 인스턴스 knee point 식별

### 5.1 Q-LOAD-4 (a) 결정 정합

본 §5 는 사용자 사전 결정 Q-LOAD-4 (a) "단일 인스턴스 한계만 식별" 의 단일 출처다.
HA / Cluster 도입은 본 PRD 범위 밖이며, 본 §5 는 knee point 식별 절차만 잠근다.

### 5.2 Knee point 의 정의

`처리량(N)` 또는 `p99(N)` 의 그래프에서 선형 영역과 비선형 영역의 전이점:

```
처리량
  ↑
  │       ╱  ← 선형 영역 (워커 추가에 비례)
  │      ╱
  │     ╱        knee point (지수 감속 시작)
  │    ╱   ╲
  │   ╱      ╲___    ← 포화 영역 (Redis 한계)
  │  ╱
  │ ╱
  └────────────────────→ N (워커 수) 또는 RPS
```

본 PRD 는 knee point 를 **선형성 SLO (`처리량(N) / (처리량(1) × N) < α`) 가 처음
위반되는 N (또는 RPS)** 로 정의한다.

### 5.3 식별 절차 (Q-LOAD-4 (a) 정합)

```
[1] LP-3 (stress) 측정 매트릭스 실행 (N ∈ {1, 2, 5, 10})
    └ `04` §2.1 매트릭스의 LP-3 행. R = stress 등급 (Q-LOAD-6 결정)
    └ §3.3 선형성(N) 계산

[2] 선형성(N) 의 N=2 / N=5 / N=10 값 확인
    └ 첫 번째 N 에서 선형성 < α (Q-LOAD-10 결정값) → knee point 후보

[3] Redis 자원 지표 확인 (Prometheus 또는 redis-cli INFO)
    └ Redis CPU 사용률 (`redis-cli INFO cpu` 또는 외부 cAdvisor)
    └ Redis 메모리 사용률 (`webhook_relay_redis_up` Gauge 외 — 본 PRD 가 측정에서만 사용)
    └ Redis 네트워크 처리량 (호스트 측 docker stats)

[4] Knee point 의 판정
    └ Redis CPU > 80% 지속 (W_load 의 90% 이상 구간) → Redis CPU bound
    └ Redis 메모리 > 80% → 메모리 bound (BullMQ 큐 길이 증가가 원인)
    └ Redis 네트워크 > 호스트 네트워크 한계의 80% → 네트워크 bound
    └ 셋 다 아니면 → 본 시스템 워커 자원 경합 (Redis 외 원인)

[5] 결과 보고서에 knee point 기록
    └ 측정 메타데이터 (`02` §5) + knee point N (또는 RPS) + bound 원인
```

### 5.4 Knee point 의 표현 형식

| 측정 | knee point N | knee point RPS (W_load 평균) | bound 원인 | LP-ID |
|------|---------------|--------------------------------|-------------|--------|
| (PLAN 측정 결과) | (PLAN) | (PLAN) | CPU / 메모리 / 네트워크 / 워커 경합 | LP-3 |

본 §5.4 는 PLAN 측정 결과로 채워질 표의 형식 잠금.

### 5.5 Knee point 식별의 한계

- **호스트 사양 한계 vs Redis 한계** — 측정 호스트의 CPU / 메모리 / 네트워크가 Redis
  보다 먼저 한계에 도달할 수 있다. 본 PRD `02` §4.2 의 cgroup 한정값 + §4.3 호환성
  메모가 호스트 한계 식별의 기초.
- **N 매트릭스의 거친 해상도** — N ∈ {1, 2, 5, 10} 의 4 점은 knee point 의 정밀
  위치 식별에 부족할 수 있다. 거친 식별 후 PLAN 단계가 미세 매트릭스 (예: N=3, 4,
  6, 7, 8) 를 추가 측정 가능.
- **시간 변동** — 측정 시점의 호스트 노이즈 / Redis 메모리 fragmentation 누적이 knee
  point 를 변동시킬 수 있음. 재측정 (`02` §7.2 ±5%) 으로 안정성 확인.

## 6. HA / Cluster 도입 트리거 조건

### 6.1 본 PRD 의 범위 (Q-LOAD-4 (a) 정합)

범위 밖 — 별도 운영 PRD 책임 (`00-overview` §3 N4.2 정합).

### 6.2 트리거 조건 형식

다음 중 하나라도 운영에서 충족되면 HA / Cluster 도입 PRD 진입이 권장된다.

| 트리거 조건 | 측정 지표 (3단계 메트릭 또는 외부 Redis 지표) | 임계 |
|--------------|---------------------------------------------|--------|
| **T1 Redis CPU 포화** | Redis CPU 사용률 (cAdvisor 또는 `redis-cli INFO cpu`) | > 80% 지속 (1시간 윈도우의 50% 이상 구간) |
| **T2 Redis 메모리 포화** | Redis 메모리 사용률 | > 80% |
| **T3 본 시스템 knee point 가 운영 부하 안에 들어옴** | §5.4 knee point RPS < 운영 평균 RPS × 2 (margin) | (운영 측정) |
| **T4 SLO-3 전달 지연 위반 지속** | 3단계 SLO-3 (`p99 ≤ 5s`) burn rate alert 14.4× 가 N 증가로도 회복 안 됨 | (`prd-phase3/04` §5.2.2) |
| **T5 SLO-4 DLQ 적재율 위반 지속** | 3단계 SLO-4 burn rate alert 가 N 증가로도 회복 안 됨 | (`prd-phase3/04` §5.2.3) |

### 6.3 트리거 후 진입할 PRD

- 단일 Redis 한계 → **Redis HA / Cluster PRD** (현재 자리 예약 없음 — 본 PRD 가 자리
  예약 제안. `05` §4 PRD 변경 제안).
- 본 시스템 자원 경합 → **본 PRD 본문 갱신** 또는 별도 PRD (워커 동시성 튜닝 /
  연결 풀 사이즈 등).

### 6.4 트리거 조건의 운영 모니터링

본 §6.4 는 운영에서 트리거 조건을 자동 감지하기 위한 **권고** 다 (본 PRD 가 새

## 7. 수용 기준 (AC)

본 §7 은 본 PRD `00-overview` §5 AC4.5 의 글자 단위 정합 절이다.

  계산을 명시.
- **AC4.5.2** §4 가 수평 확장 SLO 의 두 형태 (SLO-H-1 처리량 선형성 + SLO-H-2 p99

## 8. 불변식 (Invariants)

- **I4.19 (N 매트릭스 폐쇄성)** §2.1 의 N ∈ {1, 2, 5, 10} 은 본 PRD 가 잠그는 매트릭스.
  새 N 값 추가는 본 PRD 본문 갱신 PR.
- **I4.20 (`--scale worker=N` 의 의미 보존)** 1~2단계 결정 잠금 — N 인스턴스가 같은
  Redis 큐를 공유 소비. 본 PRD 가 N 의 의미를 변경하지 않는다. `architecture.md` §2
  컴포넌트 표 정합.
- **I4.21 (Redis 단일 인스턴스 가정 보존)** Q-LOAD-4 (a) 결정 정합. 본 PRD 측정 환경은
  Redis 단일 인스턴스. HA / Cluster 측정은 별도 PRD.
- **I4.22 (수평 확장 SLO 의 상대성)** §4.1 / §4.2 의 SLO-H-1 / SLO-H-2 는 N=1 측정값을
  기준으로 한 상대 SLO. 3단계 SLO-1~4 의 절대 임계와 보완적이며 동시 만족 필요
  (§4.5 정합).
- **I4.23 (SLO-H-1 / SLO-H-2 의 SLI 메트릭 보존)** §4.4 의 SLI PromQL 은 3단계 메트릭
  카탈로그(`prd-phase3/01` §3) 의 메트릭만 사용. 새 메트릭 도입 0건 (`00-overview`
  §2 G4.3 / §3 N4.6 정합).
- **I4.24 (HA / Cluster 도입은 별도 PRD)** §6 의 트리거 조건만 본 PRD 가 명문화.
  도입 자체는 본 PRD 범위 밖 (`00-overview` §3 N4.2 정합).

## 9. 리스크 / 오픈 퀘스천

본 PRD `00-decisions-needed.md` §🟡 의 Q-LOAD-10 / Q-LOAD-11 이 본 문서 §4 의 옵션
정리 + 결정 위임으로 잠금 대기 중. 그 외 본 문서 단위 리스크:

- **R4.14** 측정 호스트 사양 한계와 N 매트릭스의 충돌 — N=10 시 worker × 10 cgroup
  cpus 가 호스트 코어 수를 초과하면 cgroup 격리 의미 약화. `02` §4.2 메모 + 본 §2.3
  체크.
- **R4.15** Q-LOAD-10 (a) α=0.7 채택 시 본 시스템의 수평 확장 능력을 약하게만 약속 —
  포트폴리오 어필 약화 위험. (b) α=0.8 또는 (c) α=0.9 가 어필 강화.
- **R4.16** Redis 외부 메트릭(`redis_exporter`) 미도입 시 knee point bound 원인 식별
  부정확 — §5.3 [3] 단계가 외부 도구 의존. 본 PRD 가 `redis_exporter` 추가 도입을
  결정하지 않는다 (`00-overview` §3 N4.6 정합 + 새 의존성 도입 금지).
- **R4.17** LP-4 (spike) × N 매트릭스의 회복 시간 측정이 본 PRD `01` §3.1 의 잠정
  형태에서 정밀 측정 가능한지 — Q-LOAD-8 (W_cooldown) 결정과 정합 필요.
- **R4.18** N=10 측정의 docker-compose `ports: "3001:3001"` 의 충돌 — `worker` 서비스
  의 첫 인스턴스만 호스트 3001 에 매핑되고 추가 인스턴스는 동적 포트 폴백
  (`docker-compose.yml` 코멘트 정합). Prometheus scrape 는 컨테이너 network 로
  영향 없음. PLAN 단계 확인.
