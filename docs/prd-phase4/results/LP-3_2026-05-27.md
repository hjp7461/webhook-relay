### 4.1 선형성 (N=1, R=500)

```
선형성 = LP-3 처리량 (Prometheus 측 throughput avg) / 기대 처리량 (500 RPS)
       = 20.02 / 500
       = 0.040  (PLAN 임계 0.8 의 5%)
```

**knee 명백 진입** (PRD `prd-phase4/04` §5.2 + PLAN §3.1 정합).

### 4.2 p99 처리 지연 vs LP-2 nominal

- LP-2 normal SLO-3 전달 p99 = **9.91ms** (LP-2 보고서 §2.1 정합).
- LP-3 SLO-3 전달 p99 max = 1620ms / 평균 550ms.
- **배율 = 1620 / 9.91 ≈ 163× (max) / 550 / 9.91 ≈ 55× (avg)**.
- PLAN 임계 LP-2 × 1.5 = 14.87ms. **LP-3 은 임계의 ~108× (max) 명확
  초과**.

### 4.3 큐 길이 정상값 vs LP-3

- 정상 영역 (LP-2 nominal) 의 큐 waiting = 0 근처.
- LP-3 의 큐 waiting = 평균 104,800 / max 174,500 — **워커 포화 명확**.

### 4.4 bound 원인 분류 (PRD `prd-phase4/04` §5.3 [4])

본 측정의 bound 원인은 **단일 Redis 인스턴스의 RDB snapshot 시 fork()
메모리 한계** — 본 시스템의 capacity 가 워커 측 / API 측 / 네트워크 측이
아닌 **Redis 측의 fork-time 메모리 압박**에서 먼저 도달.

**증거 (api.log 1,211건 + worker.log 60건 의 동일 ReplyError 패턴):**

```
ReplyError: MISCONF Redis is configured to save RDB snapshots, but it's
currently unable to persist to disk. Commands that may modify the data set
are disabled, because this instance is configured to report errors during
writes if RDB snapshotting fails (stop-writes-on-bgsave-error option).
```

**인과 경로**:

1. R = 500 RPS × P = 64KB payload → 워커 처리 capacity (~20 RPS @
   concurrency=20) 와 차이 → waiting 큐 메모리 폭증 (1.56 GB/min).
2. ~3분 후 Redis 메모리 ~6.4GB 도달 → Docker VM 7.65GB 한계 근접.
3. Redis 가 RDB snapshot 시도 (auto-save trigger — `save 3600 1 / 300 100 /
   60 10000` default) → **fork() 시 COW 로 일시적 ~13GB 필요** → fork
   실패.
4. `stop-writes-on-bgsave-error=true` (default) 활성화 → Redis 가 **모든
   write 명령 거부** → BullMQ `queue.add` 가 거부 → API 가 500 응답.
5. 5xx 응답 100% sustained → k6 측 RPS 도달 cap (timeout 5초) + 큐는 더
   이상 자라지 못함 (write 거부) → 워커 측 처리는 진행되지만 새 요청 인입
   없음 → sustained dead-zone.

**Q-LOAD-4 (a) "Redis 단일 인스턴스 한계 식별" 정합** — 본 측정이 단일
인스턴스의 한계를 명확히 식별. 한계는 **CPU 가 아닌 메모리 (fork-time)
이며, capacity 는 ~6.4GB 메모리 (= LP-3 의 ~3분 W_load) 까지**.

> **부수 발견**: 워커 측 357건의 RetriableError `Delivery aborted (timeout)`
> 은 receiver 도 동일 api 컨테이너에서 동작 → API 가 등록 거부 모드일
> 때 receiver 도 동일 5xx 응답 / connection 끊김 → 워커 fetch 가 timeout.
> 본 부수 신호는 LP-3 의 bound 원인 (Redis fork) 의 종속 효과이며 worker
> 자체의 한계가 아니다.

### 6.1 측정의 의미

본 LP-3 측정은 **단일 Redis 인스턴스의 절대 capacity 가 R=500 RPS @
P=64KB 부하에서 ~3분 만에 도달**함을 식별. 본 capacity 는 LP-3 의 W_load
30분 전체를 견디지 못하며, capacity 의 bound 가 **CPU 가 아닌 메모리
(fork-time)**.

### 6.2 후속 작업 cross-link (M-LOAD-5 / M-LOAD-6 영역)

- **M-LOAD-5 진입 전 PRD `prd-phase4/04` §2.3 재검토 권장** — 본 측정에서
  Redis 자체엔 cgroup limit 없으나 Docker VM 7.65GB 한계가 사실상 단일
  Redis 의 한계로 작용. M-LOAD-5 의 N ∈ {1, 2, 5, 10} × LP-2 측정에서도
  동일 VM 한계가 작용할 수 있음 — 측정 호스트 의 Docker Desktop VM 메모리
  설정 재검토.
- **M-LOAD-6 의 Redis HA / Cluster 트리거 조건** (PRD §6) — 본 측정이 T1
  트리거 (단일 Redis CPU/메모리 한계 도달) 의 1차 증거. T2/T3 트리거는
  M-LOAD-5/LP-4 결과와 함께 종합 판정.
- **SLO 잠정값 재조정 (Q-LOAD-9)** — 본 측정은 R=500 stress 영역이므로
  SLO 잠정값 (R~100 nominal 영역 기준) 의 재조정 근거가 아니다. SLO 재조정
  은 M-LOAD-5 의 N=1/2/5/10 × LP-2 nominal 분포 기반 + M-LOAD-6 의 최종
  보고서 책임.

### 6.3 LP-3 재측정 권장 안 함 (사용자 결정 위임 보조 자료)

본 측정의 bound 원인이 **단일 Redis 인스턴스의 fork-time 메모리** 라는
점이 명확히 식별되었으므로, R=500 RPS @ N=1 환경에서의 재측정은 동일
패턴이 재현될 것이 예상됨 — **재측정의 추가 정보 가치 0**. M-LOAD-5 의
수평 확장 (N > 1) 측정이 본 bound 의 완화 가능성 (워커 분산 → 큐 적재
속도 감소) 을 검증할 자리.
