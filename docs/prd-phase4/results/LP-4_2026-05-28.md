    0 근처 → spike 동안 ~24,000/30s = 800 RPS 의 잉여 인입 → 워커 100 RPS

## 6. 부수 발견 + 향후 fix 권장

### 6.1 LP-4 의 register_rps_achieved query empty

본 측정의 `prom-queries.json` 의 `register_rps_achieved` (
`sum(rate(webhook_relay_api_requests_total{route="/webhooks"}[10s]))` ) 가
matrix series=0 반환. 동일 query 가 LP-3 측정에서는 정상 (n=70, avg=100.78).

가설:
- Prometheus scrape interval (보통 15s) 과 step=5s + rate window [10s] 의
  sparse 분포 — rate window 안에 최소 2 samples 필요한데 5s step 의 일부
  지점에서 미달.
- LP-3 는 step=15s 라 scrape interval 과 일치 → 정상 returned.
- 본 query 의 step 을 15s 로 늘리거나 rate window 를 30s 로 늘리면 해소
  가능성. 별도 fix commit (run-lp-4.sh line ~242: `sum(rate(... [10s]))` →
  `sum(rate(... [30s]))` 또는 step 변경).

본 query empty 가 LP-4 의 측정 의미 (회복 시간 33.1초) 에는 영향 없음 —
spike RPS 도달 검증을 k6 측 + 큐 적재 속도로 대체 가능.

### 6.2 run-lp-3.sh / run-lp-4.sh echo backtick (LP-3 보고서 §7.1 cross-link)

LP-3 측정에서 발견된 cosmetic 버그 (echo "Next steps" 안의 backtick-wrapped
path 의 shell command substitution 시도) — run-lp-4.sh 의 line ~205, 305,
335 등 동일 패턴 존재 확인. LP-4 측정 시에도 동일 stderr 출력 발생했을
가능성. 별도 fix commit 으로 처리.

### 6.3 LP-2 nominal 영역과의 정합성

LP-4 의 W_base_1 + W_base_2 의 SLI 분포가 LP-2 normal 영역과 거의 일치
(SLO-2 등록 p99 max 9.9ms vs LP-2 normal p99 4.97ms 의 ~2×, SLO-3 전달
p99 max 42.65ms vs LP-2 normal p99 9.91ms 의 ~4.3×). 차이의 대부분은 spike
구간의 burst 가 평균 통계에 영향을 미친 결과. LP-2 nominal 잠정 SLO 분포
와 본 측정의 base 분포가 정합.
