## 6. Notes — fix root cause + 1차 측정 무효 사례

### 6.1 첫 측정 (`08:01Z → 10:46Z`) 의 S3 W3 attempts ≈ 1 발견

첫 LP-2 측정 (commit `37ff8cc` HEAD) 에서 S3 변형의 W3 attempts 평균이
1.0002 로 측정 (의도 K+1=3 과 큰 괴리). PLAN §3.3 결과 무효 조건 발동.

### 6.2 Root cause 조사

다음 단계로 가설 검증 (production-flow IT + smoke 1/2):
- k6 송신 url 정합 — `[setup]` + `[VU=N ITER=0]` debug 로 `?variant=s3`
  부착 확인 ✅
- Zod `z.url()` query 보존 확인 ✅
- API/Worker container 분리 모드의 receiver 라우트 등록 동일 ✅
- receiver 진입 debug — variant=s3, query 파싱 정합 확인 ✅
- **결정적 발견:** debug log 가 **첫 8 회 요청 모두 동일 HMAC (`sha256=
  7f337d428...`)** 노출. 결정성 패딩 + (VU, ITER) 가 다른 작업도 동일
  payload 길이를 만드는 패턴이 receiver 의 HMAC 키별 카운터를 충돌시킴.

### 6.3 Fix (`ec1da6d` + `35e018c`)

- lp-2.js 의 payload 안에 `idempotencyKey` 부착 — 외부 송신 본문이 매
  작업 unique.
- receiver 의 s3 카운터 키 우선순위 — `body.idempotencyKey` > HMAC fallback.
- receiver-variants IT 에 2 case 추가 (총 7 → 9 case): body.idempotencyKey
  우선 + 같은 HMAC + 다른 body.idempotencyKey 격리 회귀 가드.
- run-lp-2.sh 에 `[5b]` 단계 추가 — api/worker container log capture.

fix 검증 — smoke 3 (W_LOAD_S=60) 에서 W3 attempts avg = 3.0 정확 일치 ✅,
worker.log: started=27,006, completed=9,002, failed=18,004 (= 9,000 × 2 회
retry).

### 6.4 단계 7 IT 가 처음 잡지 못한 이유

receiver-variants IT (commit `958b13d`) 는 매 it() 마다 **고유 HMAC 값을
명시** (`sha256=s3-counter-key-1`, ...). 결정성 패딩으로 multiple unique
작업이 동일 HMAC 를 만드는 패턴은 검증되지 않았음. fix 와 함께 추가된
신규 case (commit `ec1da6d`) 가 본 회귀 가드.
