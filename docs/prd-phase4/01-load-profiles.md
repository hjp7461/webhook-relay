# 01. Load Profiles — 부하 프로필 카탈로그 · 4가지 차원 · IT-S 매핑

> 담당 페르소나: **Performance Engineer** · SRE/Observability Lead 보조
> 본 문서는 4단계 PRD 의 부하 프로필(LP-N) 카탈로그를 단일 출처로 잠근다. 다른 4단계
> 문서(`02`~`04`)는 본 문서가 정의한 LP-ID 와 4가지 차원(RPS / 페이로드 / 등록 패턴 /
> 측정 윈도우) 을 참조한다.
> 구현 코드(k6 시나리오 스크립트, 측정 자동화) 는 본 PRD 가 승인된 뒤 **후속 PLAN
> 단계** 에서 작성한다.

---

## 1. 컨텍스트 / 배경

1~2단계 IT-S1~S7 + IT-R1 은 **단일 실행** 매트릭스로 본 시스템의 보장을 증명한다
(예: IT-S1 은 1건 등록 → 처리 → 수신자 도달). 3단계 IT-OBS-1~12 는 동일한 단일
실행 매트릭스에서 메트릭이 §5 표대로 움직이는지 단언한다.

운영은 단일 실행이 아니다. 운영자가 보는 메트릭/SLO 는 **시간에 걸친 누적 부하** 의
함수다. 본 PRD `00-overview` §1 이 명시하듯, 3단계 SLO 임계값(99.5% / 0.5s / 5s /
1%) 은 **잠정값** 이고 부하 측정 없이 채택된 숫자다(`prd-phase3/04` §3.4 +
Q-OBS-11). 잠정 임계가 운영 부하 하에서 지켜지는지를 검증하려면, 1~2단계 IT-S 의
**단일 실행 시나리오를 정량 부하 프로필로 확장** 해야 한다.

본 PRD `01` 은 그 확장 함수를 잠근다. **단일 실행 → 지속 부하** 의 변환을 카탈로그로
잠가, 후속 PLAN 단계가 본 카탈로그를 그대로 k6 시나리오로 옮기게 한다.

> 본 문서가 잠그는 것은 **부하 프로필의 형태와 4가지 차원의 정의** 이지, 정확한
> RPS 숫자 / 페이로드 분포 비율 / 측정 윈도우 길이가 아니다. 후자는 Q-LOAD-6 /
> Q-LOAD-7 / Q-LOAD-8 결정 위임 — §3 / §4 / §5 가 옵션 정리 + 결정 라인을 명시한다.

## 2. 부하 프로필의 4가지 차원

본 PRD 가 정의하는 부하 프로필(LP-N) 은 다음 4가지 차원의 **잠긴 튜플** 이다.
4가지 차원 외 새 차원(예: 멱등성 키 중복률 · TLS on/off · HTTP/2) 추가는 본 PRD
범위 밖. 추가 필요 시 본 §2 표 확장 절차로 잠금 후 카탈로그 갱신.

| 차원 | 기호 | 정의 | 단위 | 영향받는 SLI |
|------|------|------|------|---------------|
| (i) RPS 등급 | `R` | `POST /webhooks` 등록 요청의 평균 RPS | requests / second | D1 / D2 / C1 / C2 |
| (ii) 페이로드 크기 | `P` | 요청 본문 바이트 분포(고정값 또는 혼합) | bytes | D3 / W2 |
| (iii) 등록 패턴 | `T` | RPS 의 시간 함수: steady / burst / spike | (enum) | C1 / D2 / W2 |
| (iv) 측정 윈도우 | `W` | 부하 인가 + 결과 수집의 시간 길이 | seconds | (all) |

### 2.1 RPS 등급 (R) — 차원 (i)

본 PRD 는 RPS 등급을 **정성 등급** (baseline / nominal / stress / spike) 으로
정의하고, 각 등급의 정확한 숫자는 Q-LOAD-6 결정 위임이다. 정성 등급의 의미:

- **baseline** — 시스템 워밍업 + 메트릭 노출 정상화 확인용. 1~2단계 IT-S1 의
  단일 실행에 가장 가까운 부하.
- **nominal** — 운영 기대 부하의 정상 영역. SLO 잠정값이 검증되어야 하는 영역.
- **stress** — knee point 식별용. p99 / 처리량 / 큐 길이가 비선형 영역에 진입하는
  지점을 찾는다 (`04` §5).
- **spike** — 짧은 시간 안에 RPS 가 nominal → stress 의 X배 로 치솟는 패턴. C1
  `queue_depth{job_state="waiting"}` 가 일시적으로 증가했다가 회복하는 양상 측정.

### 2.2 페이로드 크기 (P) — 차원 (ii)

페이로드는 본 시스템의 핫패스 비용(HMAC 서명 + Zod 파싱 + 외부 fetch body 전송) 에
선형 영향을 준다. 본 PRD 는 페이로드 크기를 **상한 (`WEBHOOK_MAX_PAYLOAD_BYTES` =
64KB) 이하의 분포** 로 정의한다.

페이로드 후보 크기:

- **small (1KB)** — 운영 평균에 가장 가까운 크기. JSON 객체 1-2 필드.
- **medium (16KB)** — D3 `api_request_body_bytes` 히스토그램의 중간 버킷.
- **large (64KB)** — 상한 직전. SSRF/HMAC/payload 직렬화 비용 최대.

정확한 분포 비율은 Q-LOAD-7 결정 위임 — §4.

### 2.3 등록 패턴 (T) — 차원 (iii)

본 PRD 가 정의하는 등록 패턴 enum (확장 금지):

- **steady** — 측정 윈도우 전체에 걸쳐 `R` RPS 가 ±5% 분산으로 유지. p99 / 처리량 /
  카디널리티의 정상 분포 확보.
- **burst** — `R_low → R_high → R_low` 의 사각파. burst 구간 동안 큐가 일시적으로
  증가하다 워커가 따라잡는 패턴 측정. 셧다운 진행 중(IT-S7) 의 부하 변형과는
  무관(Q-LOAD-5 (a) 정합 — 정적 부하만).
- **spike** — `R_base → R_spike → R_base` 의 임펄스. spike 구간이 짧고(≤ 30s)
  진폭이 크다(≥ 5× R_base). C1 큐 길이의 회복 시간(time-to-recover) 측정.

### 2.4 측정 윈도우 (W) — 차원 (iv)

본 PRD 가 정의하는 측정 윈도우는 **워밍업(W_warmup) + 부하 인가(W_load) + 쿨다운
(W_cooldown)** 의 합이다. 워밍업 / 쿨다운은 부하 인가 외 시간이며 SLI 계산에서
제외한다. 정확한 시간 값은 Q-LOAD-8 결정 위임 — §5.

본 PRD 가 잠그는 형태:

```
|---- W_warmup ----|---------- W_load ----------|---- W_cooldown ----|
                   ^                            ^
                   부하 인가 시작              부하 인가 종료
                   Prometheus query             Prometheus query
                   시작 시점 (t_start)          종료 시점 (t_end)
```

`t_start` / `t_end` 는 결과 보고서(`02` §5) 의 메타데이터에 기록 의무.

### 2.5 4가지 차원 외 항목 (명시적 금지)

다음은 본 PRD 가 부하 프로필의 차원으로 다루지 않는다. 필요 시 별도 PRD 또는
본 §2 확장 절차로 잠금.

- **멱등성 키 중복률** — IT-S2 가 단일 실행으로 이미 검증. 부하 변형이 새 신호를
  주지 않으므로 차원 제외(Q-LOAD-5 (a) 정합).
- **TLS on/off · HTTP/2** — 본 시스템 데모는 평문 HTTP. 운영 환경 측정은 별도 PRD.
- **워커 수 N** — `01` 의 부하 프로필은 워커 수 N 에 직교한다. N 의 함수 형태 측정은
  `04` 책임. LP-N × N 매트릭스는 `04` §3 가 정의.
- **카오스(워커 강제 종료 / Redis 끊김)** — Q-LOAD-5 (a) 결정 정합. 별도 카오스
  엔지니어링 PRD.

## 3. LP-N 카탈로그

본 §3 은 본 PRD 의 단일 출처다. 후속 PLAN 단계는 본 표를 그대로 k6 시나리오 ID 로
옮긴다(LP-ID == k6 scenario name).

### 3.1 카탈로그 (잠정 형태 — Q-LOAD-6 / Q-LOAD-7 / Q-LOAD-8 결정 위임)

| LP-ID | 한 줄 | R (RPS) | P (페이로드) | T (패턴) | W (측정 윈도우) | 잠정 목적 |
|-------|-------|---------|---------------|-----------|-------------------|------------|
| **LP-1 (baseline)** | 워밍업 + 정상 메트릭 노출 확인 | (Q-LOAD-6) | small 고정 | steady | (Q-LOAD-8) | 1~2단계 IT-S1 의 부하 확장. SLI 가 정상 분포에 들어가는지 확인 |
| **LP-2 (nominal)** | 운영 기대 부하 영역 | (Q-LOAD-6) | (Q-LOAD-7 분포) | steady | (Q-LOAD-8) | SLO 잠정값(99.5% / 0.5s / 5s / 1%) 이 검증되어야 하는 영역. SLO 재조정 규칙(`03` §4) 의 기초 분포 |
| **LP-3 (stress)** | knee point 식별용 | (Q-LOAD-6) | large 고정 | steady | (Q-LOAD-8) | p99 / 처리량 / 큐 길이가 비선형 영역에 진입하는 지점 식별 (`04` §5 Redis knee point) |
| **LP-4 (spike)** | 짧은 시간 급증의 회복 측정 | base → spike → base | small 고정 | spike | (Q-LOAD-8) | C1 `queue_depth{job_state="waiting"}` 의 회복 시간 측정 |

### 3.2 RPS 등급의 잠정 후보 (Q-LOAD-6 결정 위임)

본 PRD 는 RPS 등급의 **잠정 후보 셋** 만 옵션으로 정리한다. 결정자는 사용자.

| 옵션 | LP-1 | LP-2 | LP-3 | LP-4 (base / spike) |
|------|------|------|------|----------------------|
| (a) 보수 셋 | 10 | 50 | 200 | 50 / 500 |
| (b) 중도 셋 | 10 | 100 | 500 | 100 / 1000 |
| (c) 공격 셋 | 50 | 500 | 2000 | 500 / 5000 |

**트레이드오프:**
- (a) 보수 셋 — 단일 호스트 + cgroup 격리 환경에서 안정적 측정. knee point 식별이 어려울 수 있음.
- (b) 중도 셋 — 본 PRD 가 가장 합리적이라 판단하는 잠정 기본값. 단일 호스트 + cgroup 한정값으로 도달 가능한 영역이면서 knee point 식별 여지 확보.
- (c) 공격 셋 — knee point 식별 가능성 가장 높음. 단일 호스트 + cgroup 환경의 측정 노이즈 위험(측정 자체가 호스트 한계에 도달).

**잠정 권고:** (b) 중도 셋. PLAN 진입 전 사용자 잠금.

### 3.3 LP-N 의 확장 / 추가 절차

- 새 LP-ID 를 추가하려면 본 §3.1 표에 행을 먼저 추가한다 (4가지 차원의 잠긴 튜플).
- 본 PRD `00-overview` §5 AC4.2 ("부하 프로필 카탈로그 정의") 의 잠금 대상이므로
  PLAN 진입 후 추가는 본 PRD 본문 갱신 PR 을 요구한다.
- LP-ID 의 이름 변경 / 삭제는 후속 결과 보고서의 추적성을 깨므로 **금지**. 새 ID
  추가만 허용.

## 4. 페이로드 크기 분포 (Q-LOAD-7 결정 위임)

LP-2 (nominal) 에서 사용하는 페이로드 크기 분포는 Q-LOAD-7 결정 위임이다. LP-1 /
LP-3 / LP-4 는 §3.1 표에서 고정값을 사용(small / large / small).

### 4.1 분포 옵션

| 옵션 | small (1KB) | medium (16KB) | large (64KB) |
|------|-------------|---------------|---------------|
| (a) 운영 평균 가정 | 80% | 15% | 5% |
| (b) 균등 가정 | 60% | 30% | 10% |
| (c) 페이로드 비용 강조 | 40% | 40% | 20% |

**트레이드오프:**
- (a) 운영 평균 가정 — 운영 데이터(가정) 에 가장 가까움. SLI 분포가 small 위주로 편향. D3 히스토그램의 하위 버킷이 dominant.
- (b) 균등 가정 — small/medium/large 영향을 균등하게 노출. p99 가 큰 페이로드 비용을 반영.
- (c) 페이로드 비용 강조 — HMAC 서명 / fetch body 전송 비용을 의도적으로 부각. 보수적 SLO 잠금 권장 시.

**잠정 권고:** (a) 운영 평균 가정. 본 PRD 가 데모/단일 호스트 환경이므로 운영
평균에 가까운 분포로 SLI 분포를 측정. PLAN 진입 전 사용자 잠금.

### 4.2 분포 표현 형식

k6 시나리오는 본 분포를 `executor: ramping-arrival-rate` 등에서 가중치 선택으로
표현한다. 정확한 k6 코드는 PLAN 단계. 본 PRD 는 분포 비율만 잠근다.

### 4.3 페이로드 본문 형식 (잠금)

- 본 PRD 가 사용하는 페이로드는 **결정성 패딩** (예: `{"_pad": "<repeat>"}`) 으로
  목표 바이트 크기에 맞춘다. 외부 수신자(`_demo/receiver`) 가 본문을 검사하지 않으므로 내용은 무의미.
- HMAC 서명은 본문에 의존하므로(`packages/demo/src/domain/hmac.ts`) 페이로드 본문이
  결정성이어야 측정 재현성이 보장된다.
- 무작위 페이로드(난수) 는 본 PRD 범위 밖. 측정 재현성 보호.

## 5. 측정 윈도우 (Q-LOAD-8 결정 위임)

§2.4 가 정의한 (워밍업 + 부하 인가 + 쿨다운) 구조의 정확한 시간 값.

### 5.1 윈도우 옵션

| 옵션 | W_warmup | W_load | W_cooldown | 총 길이 |
|------|----------|--------|-------------|----------|
| (a) 짧은 측정 | 60s | 5m | 30s | 약 6.5분 |
| (b) sustained 측정 | 60s | 30m | 60s | 약 32분 |
| (c) long-run 측정 | 120s | 60m | 60s | 약 63분 |

**트레이드오프:**
- (a) 짧은 측정 — CI 가능. 메트릭 분포의 분산이 큼. 회귀 가드 IT-LOAD-N (Q-LOAD-13) 도입 시 후보.
- (b) sustained 측정 — 측정 분산 안정화. 본 PRD 가 가장 합리적이라 판단하는 잠정 기본값. SLO 재조정 규칙(`03` §4) 의 기초 분포 확보.
- (c) long-run 측정 — 장시간 안정성 확인 (메모리 누수 / GC pause / Redis 메모리 fragmentation). 본 PRD 의 데모/단일 호스트 환경에서는 과잉 가능.

**잠정 권고:** LP-1 (a) 짧은, LP-2 / LP-3 (b) sustained, LP-4 (a) 짧은(spike 임펄스
+ 회복 측정). PLAN 진입 전 사용자 잠금.

### 5.2 측정 윈도우와 SLI 계산 윈도우의 정합

3단계 SLI PromQL 은 `rate(...[5m])` 형태로 5분 윈도우를 사용한다(`prd-phase3/04`
§3.1). 본 PRD 의 W_load 가 5분 이상이어야 SLI 계산이 유효하다 — 옵션 (a) 가
경계, (b) / (c) 가 안전.

W_load < 5분 옵션은 본 PRD 가 채택하지 않는다(SLI rate 윈도우 위반).

### 5.3 윈도우 외 측정 (워밍업 / 쿨다운) 의 SLI 제외

- 워밍업 구간의 메트릭은 **결과 보고서에서 제외**. JIT / V8 inline cache / Redis
  connection pool 안정화로 분산이 큼.
- 쿨다운 구간은 부하 인가 후 큐 길이 회복 / DLQ 적재 마무리를 확인하는 용도. SLI
  계산에서는 제외하되 큐 길이 회복 시간은 별도로 측정 가능.
- Prometheus query 시점은 `t_start = W_warmup 종료` / `t_end = W_load 종료` 의 두
  point 가 결과 보고서에 기록된다(`02` §5).

## 6. 1~2단계 IT-S ↔ LP-N 매핑

본 §6 은 1~2단계 IT-S 시나리오가 본 PRD 의 LP-N 부하 프로필에서 어떻게 표현되는지의
매핑표다. 후속 PLAN 단계의 k6 시나리오는 본 매핑에 따라 stub 응답을 구성한다.

| IT-S | 시나리오 (1~2단계 `prd/03` §3) | LP-N 매핑 | 스텁 / 변형 |
|------|---------------------------------|-----------|--------------|
| IT-S1 (해피패스) | 1건 등록 → 처리 → 수신자 도달 | LP-1 / LP-2 | 수신자는 항상 200. D1 / W1 / W3 등 모두 success 영역 |
| IT-S2 (멱등성) | 동일 키 N회 → 1회 처리 | (별도 부하 변형 없음) | Q-LOAD-5 (a) 정합 — 본 PRD 차원 외. 1~2단계 IT-S2 가 단일 실행으로 잠금 |
| IT-S3 (재시도+백오프) | 5xx K회 → K+1번째 200 | LP-2 (변형) | 수신자 stub 이 K회 5xx 후 200. C3 `outcome="retriable_error"` / W3 attempts 분포 측정 |
| IT-S4 (max attempts → DLQ) | 항상 5xx | LP-2 (변형) / LP-3 | 수신자 stub 이 항상 5xx. C5 `reason="max_attempts_exceeded"` / SLO-4 적재율 측정 |
| IT-S5 (4xx 즉시 DLQ) | 첫 시도 4xx | LP-2 (변형) | 수신자 stub 이 항상 4xx. C5 `reason="non_retriable"` 측정 |
| IT-S6 (stalled 회수) | 워커 강제 종료 | (LP 매핑 없음) | Q-LOAD-5 (a) 정합 — 부하 + 카오스 결합은 별도 카오스 PRD |
| IT-S6b (stalled-loss recovery) | `failed(job===undefined)` | (LP 매핑 없음) | 위와 동일 |
| IT-S7 (그레이스풀 셧다운) | SIGTERM | (LP 매핑 없음) | Q-LOAD-5 (a) 정합 — 정적 부하만 |

### 6.1 매핑되지 않은 IT-S 의 처리

IT-S2 / IT-S6 / IT-S6b / IT-S7 은 1~2단계 IT 가 이미 단일 실행으로 보장을 잠금한
시나리오이며, 본 PRD 는 부하 + 카오스 결합을 명시적 비목표로 둔다(Q-LOAD-5 (a),
`00-overview` §3 N4.3). 따라서 본 PRD 의 LP-N 카탈로그는 IT-S1 / S3 / S4 / S5 만
부하 변형으로 다룬다.

미래에 카오스 + 부하 결합 PRD 가 도입되면 본 §6 표가 그 PRD 의 진입점이 된다
(`05` §4 PRD 변경 제안 참조).

## 7. 수용 기준 (AC)

본 §7 은 본 PRD `00-overview` §5 AC4.2 의 글자 단위 정합 절이다.

- **AC4.2 (재인용)** 부하 프로필(LP-N) 카탈로그 정의. 각 LP-N 이 (i) RPS 등급 +
  (ii) 페이로드 분포 + (iii) 측정 윈도우 + (iv) 1~2단계 IT-S 매핑을 명시.
- **AC4.2.1 (본 문서 내부 AC)** §3.1 카탈로그가 4 LP-ID (LP-1~LP-4) 를 잠그고,
  각 LP-ID 가 4가지 차원의 튜플로 표현된다. Q-LOAD-6 / Q-LOAD-7 / Q-LOAD-8 결정
  잠금 전에는 본 §3 / §4 / §5 의 잠정 값이 결정 라인과 함께 명시된다.
- **AC4.2.2** §6 매핑표가 1~2단계 IT-S1 / S3 / S4 / S5 (부하 변형 대상) 와 IT-S2 /
  S6 / S6b / S7 (LP 매핑 없음) 을 모두 명시. 후자의 "매핑 없음" 사유가 Q-LOAD-5
  (a) 결정 정합 cross-link.

본 §7 AC 가 충족되어야 본 PRD `00-overview` §5 의 PRD closeout 조건이 닫힌다.

## 8. 불변식 (Invariants)

본 §8 은 본 문서 단위 불변식. 본 PRD 전체 단위 불변식은 `00-overview` 가 다룬다.

- **I4.1 (LP-N 4가지 차원 폐쇄성)** 본 PRD 가 정의하는 부하 프로필의 차원은 §2
  표의 (R / P / T / W) 4개로 닫혀 있다. 새 차원 추가는 본 PRD 본문 갱신 PR 을
  요구한다.
- **I4.2 (LP-ID 안정성)** §3.1 카탈로그의 LP-ID (LP-1~LP-4) 는 **추가는 가능하나
  삭제·이름 변경 불가**. 결과 보고서의 추적성 보호. 신규 LP-ID 는 LP-5 이상으로만
  추가.
- **I4.3 (등록 패턴 enum 폐쇄성)** §2.3 의 등록 패턴 enum (steady / burst / spike)
  은 닫혀 있다. 새 패턴 추가는 본 PRD 본문 갱신 PR.
- **I4.4 (LP-N × 카오스 분리)** Q-LOAD-5 (a) 결정 정합. 본 PRD 의 LP-N 은 정적
  부하만 표현하며, 카오스(워커 강제 종료 / Redis 끊김) 와의 결합은 별도 PRD.
- **I4.5 (페이로드 결정성)** §4.3 정합. 페이로드 본문은 결정성 패딩으로 생성하며
  난수 페이로드는 허용하지 않는다. HMAC 서명 결정성 + 측정 재현성 보호.
- **I4.6 (윈도우 ≥ SLI rate 윈도우)** §5.2 정합. W_load 는 3단계 SLI rate 윈도우
  (5분) 이상. 단, LP-1 / LP-4 의 짧은 측정 옵션 (a) 도 5분 경계 이상 유지.
