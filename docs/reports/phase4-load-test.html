### 4.1 PromQL 처리량 avg 의 ramp-up 효과

처리량 PromQL avg (6.58 / 6.68 RPS) 가 목표 10 RPS 의 66% 로 표시되는
것은 `sum(rate(...[5m]))` 의 5분 윈도우가 W_load 시작 시점부터 점진적
ramp-up 으로 누적되는 효과. 21 step 의 p99 (9.99 RPS) 가 10.000 근접
→ steady-state 처리량은 정상. 후속 LP-N (W_load 32분 sustained) 에서는
ramp-up 효과 무시 가능.

### 4.4 bound 원인 분류 (PRD `prd-phase4/04` §5.3 [4])

본 측정의 bound 원인은 **단일 Redis 인스턴스의 RDB snapshot 시 fork()
메모리 한계** — 본 시스템의 capacity 가 워커 측 / API 측 / 네트워크 측이
아닌 **Redis 측의 fork-time 메모리 압박**에서 먼저 도달.

**증거 (api.log 1,211건 + worker.log 60건 의 동일 ReplyError 패턴):**

```
ReplyError: MISCONF Redis is configured to save RDB snapshots, but it's
currently unable to persist to disk. Commands that may modify the data set
are disabled, because this instance is configured to report errors during
writes if RDB snapshotting fails (stop-writes-on-bgsave-error option).
```

**인과 경로**:

1. R = 500 RPS × P = 64KB payload → 워커 처리 capacity (~20 RPS @
   concurrency=20) 와 차이 → waiting 큐 메모리 폭증 (1.56 GB/min).
2. ~3분 후 Redis 메모리 ~6.4GB 도달 → Docker VM 7.65GB 한계 근접.
3. Redis 가 RDB snapshot 시도 (auto-save trigger — `save 3600 1 / 300 100 /
   60 10000` default) → **fork() 시 COW 로 일시적 ~13GB 필요** → fork
   실패.
4. `stop-writes-on-bgsave-error=true` (default) 활성화 → Redis 가 **모든
   write 명령 거부** → BullMQ `queue.add` 가 거부 → API 가 500 응답.
5. 5xx 응답 100% sustained → k6 측 RPS 도달 cap (timeout 5초) + 큐는 더
   이상 자라지 못함 (write 거부) → 워커 측 처리는 진행되지만 새 요청 인입
   없음 → sustained dead-zone.

**Q-LOAD-4 (a) "Redis 단일 인스턴스 한계 식별" 정합** — 본 측정이 단일
인스턴스의 한계를 명확히 식별. 한계는 **CPU 가 아닌 메모리 (fork-time)
이며, capacity 는 ~6.4GB 메모리 (= LP-3 의 ~3분 W_load) 까지**.

> **R=100 LP-2 nominal 부하가 N=1 워커의 capacity 안에 들어옴** — 단일 워커의

> SLO-H-1 의 선형성 평가가 의미를 가지려면 **부하 R 이 N=1 capacity 를
> 초과** 해야 한다.

검증:
- worker concurrency 5 × 처리 시간 평균 ~10ms ≈ **단일 워커 capacity ~500 jobs/s**
- R=100 RPS = 단일 워커 capacity 의 **20%** → N 추가의 의미 없음

### 3.2 SLO-H-1 검증의 부하 영역 의존성 (시스템적 한계)

PRD `prd-phase4/04` §I4.22 인용:
> 수평 확장 SLO 의 상대성 — §4.1 / §4.2 의 SLO-H-1 / SLO-H-2 는 N=1 측정값을
> 기준으로 한 상대값.

본 측정의 발견 — **SLO-H-1 의 의미 보존성은 부하 영역에 의존**:
- **Capacity 미달 영역 (R << N=1 capacity)**: throughput 이 R 함수로 cap → 4 N 모두 동일 → SLO-H-1 위반은 자동 (linearity = 1/N)
- **Capacity 초과 영역 (R > N=1 capacity)**: throughput 이 N 함수로 증가 → SLO-H-1 검증 의미 있음
- **Capacity 경계 영역 (R ≈ N=1 capacity)**: 부분 N 증가 영향

본 마일스톤은 PRD `prd-phase4/04` §2.1 의 잠정 합계 매트릭스에 따라 LP-2 만
사용 — R=100 LP-2 nominal 영역은 capacity 미달 영역이라 **SLO-H-1 검증
의미 영역이 아님**. SLO-H-1 의 실효 검증을 위해서는 LP-2 capacity 초과
RPS (예: R=500 LP-3 ~ R=1000) 영역에서의 N 매트릭스 측정이 필요 (M-LOAD-6
또는 후속 PRD 영역, 본 마일스톤 범위 외).

### 5.1 본 측정의 round-robin scrape 영향

- 본 측정 결과의 `throughput_completed` 가 4 N 모두 동일 (~94.1 RPS) — 이는
  **부하 영역이 capacity 미달이라 처리량이 R 함수로 cap 됨** 의 효과이며,
  round-robin scrape 의 정확도 저하는 본 결과에서 **분리 불가**.
- N=1 측정은 round-robin 없음 (단일 인스턴스) → 본 측정의 baseline 신뢰 가능.
- N=2/5/10 의 throughput 이 N=1 과 동일 ≈ "round-robin scrape 가 N 인스턴스
  의 처리량을 부분 caputure 하지만 capacity 미달이라 총합 = R" 정합.

### 5.2 SLO-H-2 의 round-robin scrape 영향

- p99 는 histogram_quantile 기반 — round-robin scrape 가 N 인스턴스 분포를

- 4 N 모두 9.92ms 동일 → 부하 영역 영향 + round-robin 영향이 분리 불가.

### 5.3 정확도 한계의 종합 판정

본 측정은 **부하 영역 의존성** + **round-robin scrape 영향** 의 합산 결과:
- SLO-H-1 위반의 본질은 **부하 영역 의존성** (capacity 미달 → linearity 1/N)

### 2.3 본질 bound = Redis fork-time 메모리 (RDB snapshot 실패)

LP-3 측정의 결정적 증거 (M-LOAD-4 보고서 §4.4 정합):

```
ReplyError: MISCONF Redis is configured to save RDB snapshots, but it's
currently unable to persist to disk. Commands that may modify the data set
are disabled, because this instance is configured to report errors during
writes if RDB snapshotting fails (stop-writes-on-bgsave-error option).
```

**인과 경로**:

1. R=500 RPS × P=64KB → 워커 처리 capacity (~20 RPS @ concurrency=20) 와 차이
   → waiting 큐 메모리 폭증 (**1.56 GB/min**).
2. ~3분 후 Redis 메모리 ~6.4GB 도달 → Docker VM 7.65GB 한계 근접.
3. Redis 가 RDB snapshot 시도 (auto-save trigger — `save 3600 1 / 300 100 /
   60 10000` default) → **fork() 시 COW 로 일시적 ~13GB 필요** → fork 실패.
4. `stop-writes-on-bgsave-error=true` (default) 활성화 → Redis 가 **모든 write
   명령 거부** → BullMQ `queue.add` 가 거부 → API 가 500 응답.
5. 5xx 응답 100% sustained → k6 측 RPS 도달 cap (timeout 5초) + 큐는 더 이상
   자라지 못함 → sustained dead-zone.

본 bound 의 본질은 **단일 인스턴스의 RDB snapshot fork-time 메모리 압박**이며,
CPU / 네트워크 / 워커 경합이 아니다. Q-LOAD-4 (a) "Redis 단일 인스턴스 한계
식별" 정합 — 단일 인스턴스의 capacity 가 **~6.4GB 메모리 (= ~3분 W_load @
R=500 RPS × P=64KB)** 까지.

  position 변동 가능.

## 4. SLO 재조정 임계 산출 (Q-LOAD-9 (a) p99 × 1.5 적용)

### 4.1 baseline 영역 채택 (사용자 결정 2026-05-28 잠금)

**채택 = nominal capacity 미달 영역만 (LP-1 / LP-2 normal / M-LOAD-5 N=1)**.

근거:
- LP-3 의 capacity 초과 영역 (5xx 100% / p99 5000ms timeout cap) 은 Redis
  fork-time 메모리 cliff — SLO baseline 으로 부적절 (시스템 한계 식별의 1차
  증거이지 정상 운영 영역이 아님).
- LP-2 S3 (백오프 누적 22.22ms) / S4 (backlog 효과 7.54ms) 는 stub variant 의
  의도된 동작 — receiver 가 강제 503/400 응답을 인가했을 때의 분포라 SLO
  baseline 으로 부적절.
- LP-4 의 spike 흡수 영역 (max p99 42.65ms) 은 spike burst (1000 RPS × 30s) 의
  큐 적재 + 해제 분포 — SLO baseline 에 포함하면 평상 운영 신호가 약화.
- nominal 영역 (capacity 미달 + steady-state) 만이 SLO 의 의미 보존 영역이며,
  본 보고서 §1.3 의 재현성 ±5% 안 확인이 신뢰성 근거.

### 4.2 SLO 재조정 임계 표 (PRD `prd-phase4/03` §5.1 형식 정합)

### 4.3 SLO 재조정의 함의

**SLO-2 / SLO-3 의 공격적 재조정 (각 ~67× / ~335×)** 이 시사하는 바:

1. **3단계 SLO 잠정값이 의도적으로 매우 보수적** (운영 부하 측정 전 채택된

   회귀 감지의 첫 신호로 작동.

## 5. SLO-H-1 / SLO-H-2 검증 결과 종합 (PRD `prd-phase4/04` §4)

### 5.1 SLO-H-1 (α=0.8 처리량 선형성) — 4 N 모두 위반

목표: `throughput(N) ≥ throughput(1) × N × 0.8` (Q-LOAD-10 (b))

baseline: throughput(1) = **94.1836 RPS** (M-LOAD-5 보고서 §3 정합)

| N | throughput(N) | target (≥) | linearity = throughput(N) / (throughput(1) × N) | SLO-H-1 |
|---|---------------|-------------|--------------------------------------------------|---------|
| 2 | 94.1005 | 150.6938 | **0.4996** | ❌ 위반 |
| 5 | 94.1536 | 376.7345 | **0.1999** | ❌ 위반 |
| 10 | 94.0634 | 753.4691 | **0.0999** | ❌ 위반 |

**위반 사유 = 부하 영역 의존성 (capacity 미달, PRD §I4.22 정합)**:

- R=100 LP-2 nominal 부하가 **N=1 워커 capacity (~500 jobs/s @ concurrency=5
  × 처리 시간 ~10ms)** 의 ~20% → 추가 워커가 idle 상태.
- linearity 가 **정확히 1/N 패턴** (N=2 → 0.5, N=5 → 0.2, N=10 → 0.1) — 부하
  영역 의존성의 결정적 signal.
- 자원 경합 / Redis 포화 / cgroup 과소 모두 아님 (M-LOAD-5 보고서 §3.1 분석
  정합).

### 5.2 SLO-H-2 (β=1.2 p99 안정성) — 4 N 모두 통과

목표: `p99(N) ≤ p99(1) × 1.2` (Q-LOAD-11 (b))

baseline: p99(1) = **9.9156ms** (worker processing duration p99, M-LOAD-5
보고서 §4 정합)

| N | p99(N) | target (≤) | ratio = p99(N) / p99(1) | SLO-H-2 |
|---|--------|-------------|-------------------------|---------|
| 2 | 9.9198ms | 11.8987ms | **1.0004** | ✅ 통과 |
| 5 | 9.9397ms | 11.8987ms | **1.0024** | ✅ 통과 |
| 10 | 9.9195ms | 11.8987ms | **1.0004** | ✅ 통과 |

**4 N 모두 통과** — 최대 ratio +0.24% (M-LOAD-5 보고서 §4 정합). N=10 의
cgroup over-commit 우려 영역 (14 컨테이너 / 12 core) 도 p99 안정.

### 5.3 SLO-H-1 / SLO-H-2 의 비대칭 결과 해석 (시스템적 한계 식별)

본 검증 결과의 비대칭성 — **SLO-H-1 4 N 위반 + SLO-H-2 4 N 통과** — 의 본질은
**부하 영역에 따른 SLO 의미 보존성 차이** (PRD §I4.22 "수평 확장 SLO 의 상대성"
시스템적 한계):

| SLO 종류 | 부하 영역 의존성 | capacity 미달 영역의 평가 |
|----------|------------------|----------------------------|
| **SLO-H-1 (처리량 선형성)** | 강함 | throughput 이 R 함수로 cap → linearity = 1/N → 자동 위반 (검증 의미 없음) |
| **SLO-H-2 (p99 안정성)** | 약함 | p99 의 절대값 비교 → capacity 미달/초과 모두에서 의미 있음 (본 측정에서 안정성 명확 확인) |

**본 측정의 의미 있는 산출물:**

- ✅ **SLO-H-2 4 N 통과** — 본 시스템이 수평 확장 시 p99 안정성을 명확히 보장
  (β=1.2 임계 안에서 ratio +0.24% — 매우 보수적). 본 결과가 본 시스템의 **수평
  확장 능력에 대한 정량적 약속의 1차 증거**.
- ⚠️ **SLO-H-1 의 의미 보존 검증은 capacity 초과 영역의 측정 필요** — 본 PLAN
  범위 외. 후속 PRD/마일스톤 영역 (§6.2 cross-link).

### 5.4 정확도 한계 (PRD `prd-phase4/04` §R4.18 정정 + M-LOAD-5 §5 정합)

M-LOAD-5 fix `db23169` 이후 `docker/prometheus.yml` 의 worker scrape 가 single
static target → `docker compose --scale worker=N` (N>=2) 시 docker compose
service-level DNS round-robin 으로 매 scrape (15s) 마다 N 인스턴스 중 1 응답
→ `rate()` PromQL 의 N 합산 정확도 저하 가능성.

**본 측정의 round-robin scrape 영향 분리 불가:**

- 4 N throughput 동일 (94.06~94.18 RPS) — capacity 미달이 본질이라 round-robin
  영향과 분리 불가능.
- N=1 측정은 round-robin 없음 (단일 인스턴스) → baseline 신뢰 가능.
- SLO-H-2 의 p99 안정성도 동일 한계 — 부하 영역 영향 + round-robin 영향이
  분리 불가. 다만 ratio +0.24% 는 두 영향 합산이 매우 작음을 시사.

**정확도 한계 개선은 본 PLAN 범위 외** — `prometheus.yml dns_sd_configs` 도입
정량화는 후속 작업 (§6.2 cross-link).
