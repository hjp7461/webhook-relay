# 01. MVP (1단계) — 작업 등록 → 워커 처리 → 최소 대시보드

> 담당 페르소나: **PM** + **Architect** + **API Designer**
> 본 문서가 정의하는 모든 동작은 코드가 아닌 **요구사항**이다. 구현은 후속 PLAN 단계.
>
> **AI 협업 5원칙(CLAUDE.md §7) 적용:** 1단계 비목표(§3)는 임의로 확장하지 않는다(원칙 3). 의문점은 §9의 오픈 퀘스천에 위임한다(원칙 4). 코딩 규약은 [`04-architecture-boundaries.md`](./04-architecture-boundaries.md) §7-A를 따른다.

## 1. 컨텍스트 / 배경

1단계는 외부 클라이언트가 HTTP API로 작업을 등록하면 워커가 비동기로 소비해 외부 URL에
페이로드를 전달하는 **해피패스**를 동작 가능한 상태로 만드는 것이다. 장애 복구(재시도, DLQ,
stalled-job 회수)는 본 단계가 아니라 [`02-resilience.md`](./02-resilience.md)에서 다룬다.

해피패스가 동작한다는 것은:

- 등록 API가 작업을 큐에 적재한다.
- 워커가 큐에서 작업을 꺼내 외부 URL로 POST 한다.
- 데모용 수신 엔드포인트가 페이로드를 받아 화면 또는 로그로 확인 가능하다.
- 최소 대시보드가 큐 상태(대기/처리/완료/실패 카운트)를 사람이 볼 수 있게 보여준다.

## 2. 목표 (Goals)

- **G1.1** `POST /webhooks` 요청을 받아 큐에 적재한다. (동기 응답: `jobId`)
- **G1.2** 워커가 큐를 소비해 외부 URL로 페이로드를 전달한다.
- **G1.3** 전달 성공/실패가 작업 상태에 반영된다 (BullMQ 상태 기반).
- **G1.4** 데모용 수신 엔드포인트(`POST /_demo/receiver`)가 도착한 페이로드를 수신·기록한다.
- **G1.5** 최소 대시보드(`GET /dashboard`)가 큐 상태 카운터를 표시한다.
- **G1.6** 모든 외부 입력은 경계에서 Zod로 파싱한다.

## 3. 비목표 (Non-Goals — 본 단계에서 다루지 않음)

- **N1.1** 멱등성 키 처리 (2단계).
- **N1.2** 재시도/DLQ (2단계).
- **N1.3** HMAC 서명 (2단계 시작 시점에 도입 — [`06`](./06-security-and-ops.md) 참조).
- **N1.4** Prometheus 메트릭/Grafana (3단계).
- **N1.5** 멀티 테넌트 권한 모델 / API 키 발급·회수 / 인증 화면. (작업 등록 API의 공유 시크릿 Bearer 검증은 M2 범위 안 — **PLAN `Q-API-1` Resolved**, `05` §4.1 참조.)
- **N1.6** 대시보드 폴리싱(테마, 인증 화면, 페이지네이션 UX).

## 4. 사용자 스토리

- **U1.1 (데모 클라이언트)** 나는 `curl`로 작업 하나를 등록하면, 즉시 `jobId`를 응답으로
  돌려받고 싶다. 그래야 비동기 처리 결과를 추적할 수 있다.
- **U1.2 (데모 클라이언트)** 등록한 작업이 처리되어 내가 지정한 URL로 페이로드가 도착했음을
  확인하고 싶다.
- **U1.3 (운영자/리뷰어)** 나는 브라우저에서 대시보드를 열어 큐에 작업이 몇 개 대기 중이고,
  몇 개 처리되었는지 한눈에 보고 싶다.
- **U1.4 (개발자)** 나는 API 요청 본문이 잘못되었을 때, 어떤 필드가 왜 거부됐는지 일관된
  에러 응답으로 확인하고 싶다.

## 5. 기능 요구사항

### F1.1 — 작업 등록 API
- 엔드포인트: `POST /webhooks` (Fastify)
- 요청 본문 스키마(상세는 [`05-api-and-contracts.md`](./05-api-and-contracts.md) §2 참조):
  - `url` (string, https/http URL)
  - `payload` (JSON 객체)
  - `idempotencyKey` (string, **2단계에서 필수화**. 1단계에서는 선택적으로 수신만, 활용은 2단계)
- 응답: `202 Accepted`, body `{ jobId: string }`
- Zod 스키마로 본문을 파싱하고, 실패 시 `400 Bad Request` + 표준 에러 응답.

### F1.2 — 큐 / 워커
- 큐 이름은 `constants.ts`에 단일 상수로 정의(매직 스트링 금지).
- `core` 패키지에 BullMQ Queue/Worker를 래핑하는 모듈을 둔다. **도메인 식별자(`webhook`, `http`) 금지**.
- `demo` 패키지가 `core`를 호출해 큐를 생성하고, `webhook-delivery` 핸들러를 등록한다.
- 워커는 작업을 받아 HTTP POST를 수행한다 (1단계에선 fetch/undici 중 표준 API 선호. 새 의존성 도입은
  금지: ioredis/BullMQ/Fastify/Zod 외 외부 HTTP 클라이언트가 필요한지 PLAN 단계에서 결정).

### F1.3 — 데모 수신 엔드포인트
- 엔드포인트: `POST /_demo/receiver`
- 수신한 본문을 메모리/Redis에 일정 개수 보관 (예: 최근 50건). 새 의존성 없이 구현.
- 단순 테스트용. **본 PRD 이외의 인증/보안 책임은 없다**.

### F1.4 — 최소 대시보드
- 엔드포인트: `GET /dashboard` (HTML, 서버 사이드 렌더링 또는 정적 HTML + fetch)
- 표시 항목: 큐의 `waiting`, `active`, `completed`, `failed` 카운트.
- 자동 새로고침(예: 2~5초 간격) 또는 수동 새로고침 중 하나로 충분. 우선 구현 단순화.
- 별도 프론트엔드 의존성 도입 금지. Fastify가 서빙하는 단일 HTML로 충분.

### F1.5 — 헬스체크
- `GET /healthz`: Redis 연결 상태를 포함한 OK/FAIL 응답.

## 6. 비기능 요구사항

| 분류 | 요구 |
|------|------|
| **성능** | 1단계 목표 부하는 없음. 다만 워커 1개로 초당 수십 건의 작업을 지연 없이 처리할 수 있어야 함 (정량 측정은 4단계). |
| **보안** | 1단계에서는 인증 없음. 단, 인증 부재가 의도된 선택임을 README/PRD에서 명시. HMAC 서명은 2단계 도입. |
| **관측성** | 구조화 로그(JSON). `jobId`, `attempt`, `idempotencyKey`(2단계부터) 필드 포함. |
| **신뢰성** | 1단계 해피패스에서도 워커가 떠 있는 한 작업이 유실되지 않아야 함 (장애 복구는 2단계). |
| **타입 안전성** | `strict: true`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`. `any` 금지. |

## 7. 수용 기준 (Acceptance Criteria)

- **AC1.1** `pnpm install && docker compose up` 후, `curl -X POST http://localhost:3000/webhooks -d '{ "url": "http://localhost:3000/_demo/receiver", "payload": {"k":"v"} }'`을 실행하면 `202`와 `jobId`를 받는다.
- **AC1.2** 위 요청 후 1초 이내에 `_demo/receiver`가 `{"k":"v"}`를 수신한 흔적이 로그/메모리/엔드포인트에 남는다.
- **AC1.3** 잘못된 본문(예: `url` 누락)은 `400`과 Zod 기반 필드별 에러 메시지를 응답한다.
- **AC1.4** `GET /dashboard`는 위 작업 직후 `completed: 1` (또는 적어도 변화한 카운터)를 보여준다.
- **AC1.5** `core` 패키지 어디에도 `webhook`, `http`, `fastify`, `payload` 같은 도메인/프레임워크 식별자가 등장하지 않는다 (Architect 자체 grep 검수).
- **AC1.6** 모든 HTTP 핸들러의 입출력 타입이 Zod 스키마로부터 추론되며, 명시된 타입과 일치한다.
- **AC1.7** 단위 테스트 + 통합 테스트(해피패스)가 통과한다.

## 8. 불변식 (Invariants)

1단계에서 깨지 말아야 할 속성:

- **I1.1** "등록된 작업은 큐에 반드시 존재한다" — API가 `202`를 응답했다면 큐 상태가
  최소 `waiting` 또는 그 후속(`active`/`completed`)에 있어야 한다. (Redis가 살아 있는 한)
- **I1.2** "도메인은 `demo`에만 산다" — `core` 패키지의 코드는 `demo`나 도메인 타입을 임포트하지 않는다 (의존 방향 단방향: demo → core).
- **I1.3** "외부 입력은 검증 전엔 신뢰하지 않는다" — Zod 파싱 이전 단계에서는 `unknown`만
  존재한다.

## 9. 리스크 / 오픈 퀘스천

- **R1.1** HTTP 클라이언트 선택 (Node `fetch`(undici 내장) vs 별도 의존성 추가). 새 의존성 추가는 §2 정책상 금지. → `07` 오픈 퀘스천에 기록.
- **R1.2** 대시보드 렌더링 전략 (서버 렌더 단일 HTML vs SSE/WebSocket). 1단계는 정적 HTML + 폴링이 디폴트.
- **R1.3** 작업 페이로드 크기 상한 (예: 64 KB). `05`의 환경변수로 노출하되 기본값은 보수적으로.

## 10. 후속 단계

- 본 단계가 통과되면 [`02-resilience.md`](./02-resilience.md)로 넘어가 멱등성/재시도/DLQ를 도입한다.
- 1단계의 어떤 결정도 **임의로** 2단계 요구사항을 침범하지 않는다. 침범이 필요하면 `07`의
  오픈 퀘스천에 사람의 결정을 요청한다.
