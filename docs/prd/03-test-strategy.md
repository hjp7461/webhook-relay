# 03. Test Strategy — 7개 핵심 시나리오의 검증 요구사항

> 담당 페르소나: **QA / Test Strategist**
> 본 문서는 구현보다 먼저 작성되는 테스트의 **요구사항**을 정의한다.
> CLAUDE.md §5에 명시된 7개 통합 시나리오를 PRD 수용 기준으로 번역한다.
> 구현 코드(테스트 코드 포함)는 본 PRD가 승인된 뒤 **후속 PLAN 단계**에서 작성한다.
>
> **AI 협업 5원칙(CLAUDE.md §7) 적용:** 본 문서가 정의하는 7개 시나리오의 테스트는 **구현보다 먼저** 작성된다(원칙 2). 시나리오에 명시되지 않은 보장은 임의로 추가하지 않는다(원칙 3). 불확실한 경계값은 §10의 오픈 퀘스천에 기록한다(원칙 4).

## 1. 컨텍스트 / 배경

이 프로젝트의 핵심 어필 포인트는 "장애 상황에서도 보장이 지켜진다"를 **재현 가능한 테스트**로
증명하는 것이다. 따라서 본 문서는 다음을 충족한다:

- 모킹된 Redis는 **신뢰하지 않는다**. 통합 테스트는 Testcontainers로 실제 Redis 컨테이너를 띄운다.
- 시간 의존 테스트(백오프 지연)는 fake timer로 결정론적으로 만든다.
- 통합 테스트는 서로 간섭하지 않도록 고유한 큐 이름/prefix를 사용한다.
- 동시성/장애 복구 로직은 **테스트가 먼저 작성**된 뒤 구현된다 (CLAUDE.md §7-2).

## 2. 테스트 분류와 경계

| 분류 | 파일 명명 | 의존성 | 책임 |
|------|-----------|--------|------|
| 단위 | `*.unit.test.ts` | Redis 없음. fake timer 가능. | 순수 로직: 백오프 계산, 멱등성 키 검증, 에러 분류 함수, Zod 스키마 |
| 통합 | `*.integration.test.ts` | Testcontainers의 실제 Redis 컨테이너. | 큐/워커 상호작용, 실패 복구, 셧다운 |

### 단위 vs 통합 결정 규칙

- "Redis가 없으면 검증이 안 되는가?" → 통합.
- "함수 입출력만으로 검증되는가?" → 단위.
- 애매하면 **통합**을 선택한다 (이 프로젝트의 약속은 보장이지, 속도가 아니다).

## 3. CLAUDE.md §5 핵심 시나리오 매핑

다음 7개 시나리오는 각각 하나 이상의 통합 테스트 케이스로 구현된다. PLAN 단계에서 아래 표의
**테스트 ID**가 그대로 `describe`/`it` 명에 등장해야 한다 (추적성).

| # | 시나리오 (CLAUDE.md §5) | 테스트 ID | 검증 포인트 | 관련 불변식 |
|---|--------------------------|-----------|-------------|-------------|
| 1 | 작업 등록 → 워커 처리 → 완료 (해피패스) | `IT-S1-happy-path` | API `202` → BullMQ 상태 `completed` → 수신자 페이로드 일치 | I1.1, I2.1 |
| 2 | 동일 멱등성 키 중복 등록 → 한 번만 처리 | `IT-S2-idempotency` | 같은 key로 N회 등록 시 핸들러 호출 횟수가 정확히 1 | I2.2 |
| 3 | 재시도 가능 에러 → 지수 백오프로 N회 재시도 | `IT-S3-retriable-backoff` | 5xx/timeout에 대해 `attemptsMade`가 단계적으로 증가하며 지연도 단계적으로 늘어남 | I2.1, I2.3 |
| 4 | 최대 재시도 초과 → DLQ로 이동 | `IT-S4-max-attempts-dlq` | `attempts` 초과 후 작업이 DLQ에서 조회 가능, 원 큐에서는 사라짐 | I2.1, I2.4 |
| 5 | 재시도 불가 에러(4xx) → 재시도 없이 즉시 격리 | `IT-S5-non-retriable-immediate-dlq` | 4xx 응답 시 `attemptsMade == 1`로 DLQ에 이동 | I2.3, I2.4 |
| 6 | 워커 강제 종료 → 다른 워커가 회수 | `IT-S6-stalled-recovery` | 처리 중 워커 `SIGKILL`(또는 강제 close) → 동일 작업을 다른 워커가 처리 | I2.5 |
| 7 | SIGTERM 수신 → 진행 중 작업 완료 후 종료 | `IT-S7-graceful-shutdown` | 진행 중 작업이 완료된 뒤에만 프로세스가 종료, 신규 작업은 받지 않음 | I2.6 |

### 추가 회귀 보호 (필수는 아님, 권장)

- `IT-R1-domain-boundary` — `core` 패키지의 소스에서 `webhook`/`http`/`fastify` 문자열이 검출되지 않는다 (정적 grep 또는 import-check 테스트). I1.2, I2.7.

## 4. 시나리오별 상세 요구사항

### IT-S1 — 해피패스
- **준비:** Testcontainers Redis, 임시 큐 이름, 데모 수신자(Fastify in-process 또는 mock 서버).
- **실행:** `POST /webhooks` 1회.
- **검증:** API 응답 `202`, 수신자에 페이로드 도착, BullMQ 상태 `completed`, 카운터 증가.
- **결정성:** fake timer 없이 실제 wait는 짧은 polling(타임아웃 5초 이내)으로.

### IT-S2 — 멱등성
- **준비:** 동일 `idempotencyKey`로 등록 요청 2~3회 동시 전송.
- **검증:** 핸들러가 정확히 1회 호출, 동일 jobId 반환, 수신자에 1건만 도착.
- **주의:** 이 테스트는 BullMQ의 `jobId` 기반 중복 방지를 사용하는지 검증 — 자체 키 저장소
  구현을 도입하지 않아야 함 (`02` §F2.1).

### IT-S3 — 재시도 가능 에러 + 백오프
- **준비:** 수신자가 처음 K번은 5xx, K+1번째에 200을 반환하도록 스텁.
- **실행:** 작업 등록 → fake timer로 백오프 시각을 점프.
- **검증:**
  - 시도 횟수가 단계적으로 증가
  - 각 시도 간 지연이 지수 백오프 공식과 일치 (허용 오차 포함)
  - 최종적으로 `completed`
- **결정성:** Vitest fake timer 사용. 실제 wall-clock 의존 금지.

### IT-S4 — 최대 재시도 초과 → DLQ
- **준비:** 수신자가 항상 5xx.
- **실행:** `WEBHOOK_MAX_ATTEMPTS`에 도달할 때까지 진행.
- **검증:** DLQ에 작업이 적재되어 있고, 페이로드와 마지막 에러 컨텍스트가 보존됨. 원 큐에는 없음.

### IT-S5 — 재시도 불가 에러 즉시 격리
- **준비:** 수신자가 첫 시도에서 4xx.
- **검증:** `attemptsMade == 1`로 DLQ 이동. 재시도가 발생하지 않음 (시간 점프 후에도 추가 호출 없음).

### IT-S6 — Stalled-job 회수
- **준비:** 워커 A와 워커 B 두 인스턴스를 띄움. A가 작업을 잡고 처리 도중 강제 종료
  (워커 인스턴스의 `close({ force: true })` 또는 프로세스 종료 시뮬레이션).
- **검증:** `stalledInterval` 이후 B가 같은 작업을 처리. 작업이 유실되지 않음.
- **주의:** 본 테스트는 wall-clock 의존성이 불가피한 부분이 있으므로, `stalledInterval`을
  **테스트용으로 짧게** 설정한다(예: 500ms). 환경변수 또는 옵션 주입으로 제어.

### IT-S7 — 그레이스풀 셧다운
- **준비:** 핸들러가 의도적으로 1~2초간 처리하도록 구성. 작업 1건 등록 후 워커에 SIGTERM 송신.
- **검증:**
  - 작업이 `completed` 상태로 끝남 (강제 중단되지 않음)
  - 셧다운 도중 등록된 신규 작업은 받지 않음 (워커가 paused 또는 closed)
  - 셧다운 타임아웃 내에 프로세스 종료

## 5. 단위 테스트 요구사항

다음은 단위 테스트로 충분하며, 반드시 존재해야 한다.

- **UT-1** 백오프 지연 계산 함수: `(attempt, baseMs) -> delayMs` 공식의 결정성·경계값.
- **UT-2** 에러 분류 함수: HTTP 상태/네트워크 에러 입력 → `RetriableError`/`NonRetriableError` 매핑.
- **UT-3** 작업 등록 요청 Zod 스키마: 필수 필드 누락·잘못된 URL·페이로드 크기 한도 검증.
- **UT-4** 환경변수 Zod 스키마: 기본값/타입 강제.
- **UT-5** 멱등성 키 정합성 검증 함수(형식·길이 제한).
- **UT-6** HMAC 서명 생성 함수의 결정성(같은 입력 → 같은 서명).

## 6. 테스트 인프라 / 도구

- **Vitest** — 단일 테스트 러너.
- **Testcontainers** — Redis 컨테이너 lifecycle 관리.
- **Fake timer** — 시간 의존 테스트의 결정성. 통합 테스트에서도 BullMQ가 허용하는 한 사용.
- **데모 수신자(mock server)** — `demo` 패키지 내부에서 Fastify in-process 서버 또는 가벼운 stub로 구현. 새 의존성 추가 금지.

> **금지 사항:** 모킹된 Redis 라이브러리(`ioredis-mock` 등)는 사용하지 않는다. CLAUDE.md §5
> "모킹된 Redis는 신뢰하지 않는다"를 그대로 따른다.

## 7. 비기능 요구사항 (테스트 자체에 대한)

- **격리:** 각 테스트는 고유한 큐 이름/prefix를 쓴다. 공유 상태 금지.
- **속도:** 통합 테스트 1건의 최대 실행 시간은 30초를 넘기지 않는다 (스털드 회수 등 wall-clock
  의존 항목은 임계값을 짧게 설정).
- **결정성:** flaky test 금지. 통합 환경에서도 5회 연속 실행 시 5회 통과해야 한다.
- **CI 적합성:** `pnpm test:unit`은 Docker 없이 실행 가능. `pnpm test:integration`은 Docker 데몬을 요구한다.

## 8. 수용 기준 (Acceptance Criteria)

- **AC3.1** 위 7개 통합 시나리오에 대응하는 테스트 ID가 그대로 코드에 존재한다.
- **AC3.2** 단위 테스트 UT-1~UT-6이 모두 존재하며 통과한다.
- **AC3.3** 테스트 파일 명명 규칙(`*.unit.test.ts`, `*.integration.test.ts`)이 일관된다.
- **AC3.4** 통합 테스트 디렉터리 내에서 `ioredis-mock` 같은 Redis 모킹 라이브러리가 등장하지 않는다.
- **AC3.5** 시간 의존 테스트가 fake timer로 결정론적으로 작성되어 있다.

## 9. 불변식 / 원칙

- **TI-1** 보장에 관한 테스트는 모킹이 아니라 실제 Redis 컨테이너로 검증한다.
- **TI-2** 시나리오 → 테스트 → 불변식의 매핑이 본 문서 §3에 명시되어 있고, 코드에서 동일한 ID가
  추적된다.
- **TI-3** 테스트가 깨졌을 때 코드를 고치지, 테스트를 약화시키지 않는다.

## 10. 리스크 / 오픈 퀘스천

- **R3.1** Testcontainers의 컨테이너 기동 시간이 CI에서 임계가 될 경우의 워크어라운드 → `07`.
- **R3.2** `stalledInterval`을 테스트용으로 얼마나 짧게 줄여도 되는가 (BullMQ의 안전 임계) → `07`.
- **R3.3** SIGTERM 핸들링 통합 테스트에서 자식 프로세스 vs 동일 프로세스 워커 선택 → `07`.
