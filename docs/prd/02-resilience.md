# 02. Resilience (2단계) — 멱등성 · 재시도 · DLQ · stalled-job 회수

> 담당 페르소나: **Architect**(불변식) + **QA/Test Strategist**(검증) + **SRE(범위 제한)**
> 본 단계의 모든 동작은 [`03-test-strategy.md`](./03-test-strategy.md)의 7개 시나리오로 **테스트가 먼저 작성**된 뒤 구현된다.
> 구현 코드는 본 PRD가 승인된 뒤 **후속 PLAN 단계**에서 작성한다. 본 문서는 코드를 작성하지 않는다.
>
> **AI 협업 5원칙(CLAUDE.md §7) 적용 메모:**
> - (1) 본 단계의 불변식은 사람이 §8에 먼저 정의한다.
> - (2) §G2.7 — 모든 케이스가 통합 테스트로 증명된다(테스트 우선).
> - (3) §3 비목표를 임의 확장하지 않는다.
> - (4) 불확실하면 §9 리스크/오픈 퀘스천에 선택지로 기록한다.
> - (5) 위반 코드 발견 시 임의 수정 금지, 사람에게 보고.

## 1. 컨텍스트 / 배경

1단계가 "동작한다"를 보였다면, 2단계는 "장애 상황에서도 보장이 깨지지 않는다"를 증명한다.
다음 장애 케이스를 전제로 한다.

- 외부 수신자가 일시적으로 실패한다 (네트워크, 5xx, 타임아웃).
- 외부 수신자가 영구적으로 거부한다 (4xx).
- 클라이언트가 같은 작업을 여러 번 등록한다 (재전송, 우발적 중복).
- 워커가 처리 중 크래시한다 (OOM, SIGKILL).
- 운영자가 워커를 의도적으로 종료한다 (SIGTERM, 배포).

본 단계는 이 모든 케이스에 대해 **명시적 불변식**과 **재현 가능한 통합 테스트**를 둔다.

## 2. 목표 (Goals)

- **G2.1** 동일 멱등성 키로 들어온 작업은 한 번만 처리한다.
- **G2.2** 재시도 가능 에러는 지수 백오프로 최대 `N`회 재시도한다 (`N`은 환경변수로 노출).
- **G2.3** 재시도 불가 에러(4xx)는 재시도하지 않고 즉시 DLQ로 격리한다.
- **G2.4** 최대 재시도를 초과한 작업은 DLQ로 이동한다.
- **G2.5** 워커가 처리 중 강제 종료되면 다른 워커가 일정 시간 내에 작업을 회수한다 (stalled-job 회수).
- **G2.6** SIGTERM 수신 시 진행 중 작업을 마치고 그레이스풀 종료한다 (셧다운 상세는 [`06`](./06-security-and-ops.md)).
- **G2.7** 위 모든 케이스가 통합 테스트(실제 Redis 컨테이너)로 증명된다.

## 3. 비목표 (Non-Goals)

- **N2.1** DLQ 작업의 자동 재투입(재처리 자동화). 본 단계는 DLQ에 **격리만** 한다.
- **N2.2** Poison message 자동 탐지/패턴 분석.
- **N2.3** 우선순위 큐, 지연 작업 스케줄러.
- **N2.4** 메트릭/대시보드 시각화 (3단계).
- **N2.5** 분산 트랜잭션, exactly-once 시도.

## 4. 시나리오 (사용자/시스템 스토리)

각 시나리오는 [`03-test-strategy.md`](./03-test-strategy.md)의 통합 테스트와 1:1 매핑된다.

- **S2.1** 클라이언트가 동일한 `idempotencyKey`로 작업을 두 번 등록하면, 워커는 한 번만 실행한다.
- **S2.2** 수신자가 5xx를 반환하면, 워커는 백오프 후 재시도하고 결국 성공/DLQ 격리에 도달한다.
- **S2.3** 수신자가 4xx를 반환하면, 워커는 재시도 없이 즉시 DLQ로 격리한다.
- **S2.4** 최대 재시도를 초과하면, 작업은 DLQ로 이동하고 원 큐에서는 사라진다.
- **S2.5** 워커가 처리 도중 `SIGKILL`로 죽으면, 일정 시간 후 다른 워커가 같은 작업을 회수해 처리한다.
- **S2.6** 워커가 `SIGTERM`을 받으면 진행 중 작업을 끝까지 처리한 뒤 종료한다.
- **S2.7** 1단계 해피패스가 본 단계 도입 이후에도 깨지지 않는다 (회귀 방지).

## 5. 기능 요구사항

### F2.1 — 멱등성
- 작업 등록 API는 `idempotencyKey` 필드를 **필수**로 받는다 (1단계의 선택 → 2단계의 필수).
- 동일 키로 들어온 후속 요청은 기존 `jobId`를 그대로 반환한다 (HTTP `202` + 동일 jobId, 또는
  `409` 의미상 멱등 응답). 두 옵션 중 선택은 `07` 오픈 퀘스천. **단**, 실제 작업이 두 번 큐에
  적재되거나 두 번 실행되어선 안 된다.
- BullMQ의 `jobId` 기반 중복 방지를 활용한다 (별도 자체 키 저장소를 만들지 않는다 — 새 의존성/모듈 금지).
- 멱등성 키 보존 기간은 BullMQ 작업 보관 정책에 종속된다 (값은 환경변수 — `05`).

### F2.2 — 에러 분류
- `core`는 두 종류의 에러를 구분한다 (도메인 비의존):
  - `RetriableError` — 재시도해야 한다 (워커는 throw로 BullMQ에 위임).
  - `NonRetriableError` — 재시도하지 말고 격리해야 한다.
- `demo`는 도메인 에러(`WebhookDeliveryError`)를 정의하고, HTTP 응답 코드와 네트워크 에러에 따라
  위 두 클래스 중 하나에 매핑한다.
- **분류 규칙:**
  - 4xx 응답 → `NonRetriableError`.
  - 5xx 응답, 네트워크 단절, 타임아웃 → `RetriableError`.
  - 3xx는 핸들러에서 정책 결정 (기본은 `NonRetriableError` — `07` 오픈 퀘스천).

### F2.3 — 지수 백오프 재시도
- BullMQ의 `attempts` + `backoff: { type: 'exponential', delay: BASE }` 사용. 자체 구현하지 않는다.
- 파라미터(환경변수 노출 — `05` 참조):
  - `WEBHOOK_MAX_ATTEMPTS` — 기본 5
  - `WEBHOOK_BACKOFF_BASE_MS` — 기본 1000 (1초)
- 백오프 지연은 **결정론적 테스트**를 위해 fake timer로 검증한다 (CLAUDE.md §5 정책).

### F2.4 — DLQ
- DLQ는 별도의 BullMQ 큐로 구현한다 (이름은 `constants.ts`). 자체 자료구조 금지.
- DLQ 이동 트리거:
  - 재시도 가능 에러 + `attemptsMade >= attempts` 초과
  - 재시도 불가 에러 즉시
- DLQ에 들어간 작업은 **원 큐에서 사라지고**, DLQ에서는 페이로드와 마지막 에러 컨텍스트
  (분류, 응답 상태, 시도 횟수)가 보존된다.
- DLQ 작업의 **자동 재투입은 금지** (본 단계 비목표).

### F2.5 — Stalled-job 회수
- BullMQ의 `stalledInterval`, `maxStalledCount` 설정을 노출한다.
- 워커가 처리 도중 죽으면, BullMQ가 stalled 상태로 마킹하고 다른 워커가 회수한다.
- 회수 후 재시도는 §F2.3의 백오프 정책을 따른다.
- 본 PRD는 **자체 stalled-job 매니저를 구현하지 않는다**. BullMQ 메커니즘에 의존한다.
- **stalled-loss recovery (best-effort):** BullMQ 의 'failed' 이벤트가 `job === undefined`
  로 발화되는 케이스(메타데이터 복구 실패, lock 손실 등)에서 페이로드 손실을 막기 위해
  워커 프로세스 내부에 in-memory `activeJobs` 맵을 유지한다. 후보(들)를 보수적으로
  `stalled-loss-recovered` 메시지(class=Retriable)로 DLQ 에 적재한다. 회귀 단언은
  통합 테스트 IT-S6b 가 담당한다. 구현: `packages/core/src/worker.ts`.

### F2.6 — 그레이스풀 셧다운(요약)
- SIGTERM 수신 시:
  1. 새 작업 수신 중단 (워커 `pause` 또는 `close({ force: false })`)
  2. 진행 중 작업의 `await` 완료 대기 (타임아웃 환경변수)
  3. Redis 연결 정리 후 프로세스 종료
- 상세는 [`06-security-and-ops.md`](./06-security-and-ops.md) §6.

## 6. 비기능 요구사항

| 분류 | 요구 |
|------|------|
| **신뢰성** | 워커 강제 종료 후 회수까지의 최대 시간이 환경변수로 설정 가능. 기본 60초 이내. |
| **관측성** | 구조화 로그에 `attempt`, `errorClass`, `httpStatus` 포함. (상세 메트릭은 3단계) |
| **보안** | HMAC 서명 적용 ([`06`](./06-security-and-ops.md) §2). 시크릿은 로그 금지. |
| **결정성** | 백오프 지연 테스트는 fake timer로 결정론적이어야 한다. |
| **격리** | 통합 테스트는 고유한 큐 이름/prefix를 써서 서로 간섭하지 않는다. |

## 7. 수용 기준 (Acceptance Criteria)

- **AC2.1** [`03`](./03-test-strategy.md) §3의 7개 통합 시나리오가 **모두 통과**한다.
- **AC2.2** `idempotencyKey`가 누락된 작업 등록 요청은 `400`을 응답한다.
- **AC2.3** DLQ로 이동한 작업은 원 큐에서 조회되지 않으며, DLQ 큐에서 조회 가능하다.
- **AC2.4** `core/retry.ts`는 도메인 식별자를 포함하지 않는다 (Architect grep 검수).
- **AC2.5** 재시도 가능/불가 에러가 단위 테스트로 분류 함수 단위에서 검증된다.

## 8. 불변식 (Invariants) — **본 단계의 핵심**

다음은 어떤 상황에서도 깨지지 말아야 할 속성이다. PLAN 단계의 테스트는 이 불변식을
직접적으로 검증해야 한다.

- **I2.1 (At-least-once)** API가 `202`를 응답했다면 작업은 **최소 한 번** 실행된다 (성공 또는 DLQ 격리).
- **I2.2 (멱등성 흡수)** 동일 `idempotencyKey`로 들어온 작업은 큐 적재/실행 횟수가 **정확히 1**이다.
- **I2.3 (재시도 분류 결정성)** 동일한 입력(에러 타입·HTTP 상태)에 대해 분류 결과는 항상 같다.
- **I2.4 (DLQ 단방향)** 본 단계에서 DLQ에 들어간 작업은 자동으로 원 큐로 돌아가지 않는다.
- **I2.5 (회수 보장)** 처리 도중 워커가 죽어도 작업은 큐에서 영구히 사라지지 않는다.
- **I2.6 (그레이스풀 종료의 약속)** SIGTERM 수신 후 진행 중 작업은 강제 중단되지 않는다 (셧다운 타임아웃 내).
- **I2.7 (도메인 경계)** `core`의 어떤 모듈도 `webhook`, `http`, `Fastify`를 임포트하지 않는다.

## 9. 리스크 / 오픈 퀘스천

- **R2.1** 멱등성 중복 요청에 대한 응답 코드 의미(`202` 동일 jobId vs `409 Conflict`) → `07`.
- **R2.2** 3xx 응답의 분류 정책 → `07`.
- **R2.3** DLQ에서 수동 재투입 UX (이번 단계에선 미구현이지만, 미래를 위한 인터페이스 스텁을
  남길지 여부) → `07`.
- **R2.4** stalled-job 회수 임계값(`stalledInterval`)의 기본값 → `07`.
- **R2.5** "재시도 불가" 분류에서 timeout과 4xx의 경계 (예: 408 Request Timeout) → `07`.

## 10. 후속 단계

- 본 단계 통과 이후 3단계(관측성, Prometheus/Grafana)로 진행한다. 본 단계의 구조화 로그
  필드(`attempt`, `errorClass`, `httpStatus`)는 3단계 메트릭의 라벨 후보가 된다.
- 4단계(부하 테스트, 그레이스풀 셧다운 측정)는 본 단계의 불변식이 모두 통과한 다음에만 시작한다.
