# 00. Overview — Reliable Webhook Queue

> 담당 페르소나: **PM(총괄)** · Architect 보조
> 본 문서는 PRD 묶음의 진입점이다. 구현 코드는 후속 PLAN 단계에서 작성한다.

## 1. 컨텍스트 / 배경

외부 서비스로 웹훅을 안정적으로 전달하는 일은 단순한 HTTP 호출처럼 보이지만, 실제로는
다음과 같은 보장이 필요한 **신뢰성 있는 작업 큐 문제**다.

- 수신자가 일시적으로 응답하지 않을 때 (네트워크 단절, 5xx, 타임아웃) 재시도해야 한다.
- 수신자가 영구적으로 거부할 때 (4xx) 즉시 격리해야 한다.
- 워커가 처리 중에 죽어도 작업이 유실되어선 안 된다.
- 동일한 작업이 두 번 등록되어도 두 번 처리되어선 안 된다 (멱등성).
- 운영자는 큐 길이·실패율·DLQ 적재량을 실시간으로 관측할 수 있어야 한다.

본 프로젝트는 위 보장을 **검증된 라이브러리(BullMQ + ioredis) 위에서 올바르게 조립**하는
판단력을 보여주는 포트폴리오 데모다. "가장 빠른 큐"를 만드는 것이 아니라 "프로덕션급
보장을 어떻게 설계·검증하는가"를 증명한다.

## 2. 목표 (Goals)

본 PRD(1~2단계)의 목표:

- **G1.** 외부 클라이언트가 HTTP API로 웹훅 작업을 등록하면, 워커가 비동기로 소비해
  지정된 URL로 전달한다. (1단계 해피패스)
- **G2.** 동일한 멱등성 키로 들어온 중복 작업은 단 한 번만 처리한다. (2단계)
- **G3.** 재시도 가능 에러는 지수 백오프로 N회까지 재시도하고, 최대 시도 초과 시 DLQ로
  격리한다. 재시도 불가 에러(4xx)는 즉시 격리한다. (2단계)
- **G4.** 워커가 처리 중 강제 종료되어도 작업이 유실되지 않고 다른 워커가 회수한다. (2단계)
- **G5.** SIGTERM을 수신한 워커는 진행 중 작업을 마치고 그레이스풀하게 종료한다. (2단계)
- **G6.** CLAUDE.md §5의 7개 통합 시나리오가 **테스트로** 증명된다(테스트 우선).
- **G7.** 모든 외부 입력은 경계에서 Zod로 파싱되며, 도메인(웹훅) 로직은 `demo` 패키지에만
  존재한다 (`core`는 도메인 비의존).

## 3. 비목표 (Non-Goals)

본 PRD에서 명시적으로 **다루지 않는 것**:

- **N1.** Prometheus/Grafana 대시보드 구성 (3단계 — 후속 PRD).
- **N2.** 부하 테스트, 처리량·지연 분포 벤치마크, 수평 확장 측정 (4단계 — 후속 PRD).
- **N3.** Raw Redis Streams로 큐 내부를 직접 구현하는 **부록 트랙**
  (`packages/streams-internals/`는 자리만 예약. 본 PRD 범위에서 일체 손대지 않는다).
- **N4.** Exactly-once 전달 보장 (at-least-once + 애플리케이션 멱등성 채택).
- **N5.** 멀티 테넌트 권한 모델, API 키 발급/회수 같은 본격적인 인증/인가
  (필요 시 단일 공유 시크릿으로 최소화. 자세한 결정은 `07`의 오픈 퀘스천 참조).
- **N6.** 웹훅 수신자 등록부(엔드포인트 카탈로그, 구독 모델). 작업 등록 시 URL을 직접 받는다.
- **N7.** 다중 큐/라우팅, 우선순위 큐, 지연 작업 스케줄러 같은 고급 큐 기능.
- **N8.** UI 폴리싱(인증 화면, 사용자 관리 등). 대시보드는 디버그용 **최소 화면**만.

## 5. 성공 지표

본 PRD의 성공은 "지표 수치"가 아니라 **검증 가능한 도달 상태**로 정의한다.

- **S1.** `pnpm test:unit`와 `pnpm test:integration`이 모두 통과한다.
- **S2.** CLAUDE.md §5의 7개 시나리오 각각에 대응하는 통합 테스트가 **개별 파일/케이스로** 존재한다 (테스트 ↔ 시나리오 1:1 매핑이 `03-test-strategy.md`에 표로 정리됨).
- **S3.** `curl`로 `POST /webhooks` → 데모 수신 엔드포인트가 페이로드를 받는 해피패스가
  로컬 `docker compose up` 후 작동한다.
- **S4.** 워커 컨테이너를 임의로 `kill`해도 작업이 유실되지 않고, 다른 워커가 처리한다
  (통합 테스트로 증명됨).
- **S5.** `core` 패키지에 `webhook`·`http`·`fastify` 같은 도메인 식별자가 등장하지 않는다
  (Architect 리뷰 시 grep으로 확인).

## 6. 트랙 구분 (CLAUDE.md §1 반영)

- **메인 트랙 — 본 PRD 범위:** BullMQ 기반 웹훅 전송 데모 + 최소 대시보드.
  본 PRD는 메인 트랙의 1~2단계만 다룬다.
- **부록 트랙 — 본 PRD 범위 밖:** Raw Redis Streams 학습용 모듈
  (`packages/streams-internals/`). 메인 트랙이 안정화되기 전에는 착수하지 않는다.

> 본 PRD의 모든 요구사항은 메인 트랙에만 적용된다. 요청이 명시적으로 부록 트랙을 가리키지
> 않는 한, 어떤 작업도 부록 트랙으로 확장하지 않는다.

## 7. 용어집 (Glossary)

| 용어 | 정의 |
|------|------|
| **작업(Job)** | 큐에 등록되어 워커가 소비하는 단위. 본 프로젝트에서는 "외부 URL로 페이로드를 전달하는 일". |
| **멱등성 키(Idempotency Key)** | 동일 작업의 중복 등록을 방지하는 식별자. 클라이언트가 제공. |
| **재시도 가능 에러(Retriable Error)** | 일시적이라 다시 시도하면 성공할 수 있는 에러(네트워크 단절, 타임아웃, 5xx). |
| **재시도 불가 에러(Non-Retriable Error)** | 다시 시도해도 같은 결과가 나오는 에러(4xx). 즉시 격리. |
| **DLQ (Dead-Letter Queue)** | 최대 재시도를 초과했거나 재시도 불가로 분류된 작업이 격리되는 큐. |
| **Stalled Job** | 워커가 가져갔지만 ACK도 실패도 보내지 않은 채 일정 시간이 지난 작업. 다른 워커가 회수한다. |
| **불변식(Invariant)** | 시스템이 어떤 상황에서도 깨지 않아야 하는 속성. 동시성/장애 복구 설계의 출발점. |
| **그레이스풀 셧다운** | SIGTERM 수신 시 새 작업을 받지 않고 진행 중 작업을 마친 뒤 종료하는 동작. |
| **`core` 패키지** | 도메인(웹훅) 비의존 큐 추상화. BullMQ 래퍼·재시도·DLQ·셧다운 정책. |
| **`demo` 패키지** | 웹훅 전송 도메인 로직, Fastify API, 데모 수신 엔드포인트, 대시보드. |

## 8. 본 PRD의 운영 원칙

- **테스트 우선:** 동시성·장애 복구 시나리오는 구현 전 테스트가 먼저 작성된다 (`03` 참조).
- **경계에서 검증:** HTTP body, 환경변수, Redis에서 읽은 payload는 모두 Zod로 파싱한다.
- **에러 분류 명시:** 재시도 가능/불가는 핸들러 안에서 명시적으로 결정하며, 조용한 catch는 금지.
- **구조화 로깅:** `jobId`·`attempt`·`idempotencyKey`를 항상 컨텍스트에 포함한다.
  민감 정보(웹훅 시크릿, 인증 헤더)는 로그에 남기지 않는다.
- **부록 트랙 보호:** `packages/streams-internals/`는 어떤 작업에서도 임포트하지 않는다.

- 작업 등록 API 인증 방식 → **PLAN `Q-API-1` Resolved**: 공유 시크릿 Bearer(`API_BEARER_TOKEN`, 최소 32 bytes).
- HMAC 서명 알고리즘/헤더 명세 → PRD `06` §3 고정(HMAC-SHA256), replay 방어는 **PLAN `Q-SEC-2` Resolved**(본 PRD 미적용).

- DLQ 재투입 UX → **PLAN `Q-DLQ-1` Resolved**: 본 PRD에서는 격리만 수행. 재투입은 후속 PRD.
