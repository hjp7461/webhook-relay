# 11. Risks & Rollback — 가장 깨지기 쉬운 부분 + 롤백 전략

> 본 문서는 본 PLAN을 실행하는 동안 가장 자주 어긋날 수 있는 지점과, 어긋났을 때 어떻게 되돌릴지를 명시한다.
>
> **AI 협업 5원칙 적용:** 본 문서는 "불확실하면 멈추고 묻는다"(원칙 4)의 실행 가이드. 아래 리스크가 현실화되면 임의로 PLAN을 확장하지 말고 사람에게 보고한다.

---

## 1. 리스크 등급 정의

| 등급 | 의미 | 대응 |
|------|------|------|
| ★★★ | 마일스톤 차단 가능성 매우 높음 | 사전 PoC 1~2시간 필요. 결정 보류 항목과 직결. |
| ★★☆ | 마일스톤 일정 영향 큼 | 발견 즉시 사람에게 보고. PLAN 변경 검토. |
| ★☆☆ | 마일스톤 내 해결 가능 | 모니터링만. |

## 2. 핵심 리스크 목록

### R-001 — Vitest fake timer ↔ BullMQ 백오프 상호작용 (★★★)
- **대상 마일스톤:** M4 (`IT-S3`).
- **현상:** BullMQ 내부의 setTimeout/setImmediate/Redis delayed 큐가 Vitest fake timer로 완전히 가로채지지 않을 수 있다. fake timer를 켜면 BullMQ 워커가 hang 또는 시간이 진행되지 않아 테스트가 timeout으로 실패.
- **사전 신호:**
  - BullMQ 이슈 추적에서 "vitest fake timer" 검색 결과가 다수.
  - 백오프 단언이 의도와 다른 결과를 일관되게 보임.
- **대응 (우선순위 순):**
  1. **Plan A (권장 1차):** `WEBHOOK_BACKOFF_BASE_MS`를 매우 짧게(50~100ms) 설정 + wall-clock + 짧은 polling으로 측정. fake timer 미사용. 단언은 시간 비율 단언(예: attempt 2의 지연이 attempt 1의 약 2배).
  2. **Plan B:** Vitest의 `vi.useFakeTimers({ toFake: [...] })`로 BullMQ가 의존하는 타이머만 가짜로 만든다. 부작용 적은 부분 fake timer.
  3. **Plan C:** Worker 옵션의 `delay` 동작을 추상화한 후 테스트에서 직접 호출(BullMQ를 우회 — **금지**, 자체 큐 재구현 금지 정책 위반).
- **롤백 트리거:** Plan A/B 모두 1일(8시간) 내 안정화 실패 시.
- **롤백 동작:** `IT-S3`의 검증 포인트를 "1회 재시도 후 성공" 수준으로 약화하고, 백오프 공식 단위 검증(`UT-1`)에 의존도를 옮긴다. PRD `02` §F2.3 / `03` §3 IT-S3의 요구 약화는 **사람의 결정**이 필요.

### R-002 — Testcontainers 컨테이너 기동 시간 (★★☆)
- **대상 마일스톤:** M2 이후 모든 통합 테스트.
- **현상:** 로컬 환경에서 매 통합 테스트마다 Redis 컨테이너 기동에 5~15초 소요. 누적 시 PRD `03` §7의 "1건 30초 이내" 임계를 깸.
- **대응:**
  1. **공용 컨테이너 + 격리 prefix:** 컨테이너를 테스트 파일 단위가 아닌 **테스트 스위트 단위**로 공유. Redis 큐 prefix로 격리(BullMQ는 `prefix` 옵션 제공).
  2. 또는 `vitest.config.ts`의 `pool: 'forks'` + `globalSetup` 활용으로 컨테이너 1개를 모든 통합 테스트가 공유.
  3. CI 임계 시 Q-OPS-1 (b) 잡 분리로 처리.
- **롤백 트리거:** 로컬에서 통합 테스트 1회 풀런이 5분을 초과.
- **롤백 동작:** Plan 1/2 적용. 그래도 안되면 IT-S6 등 wall-clock 의존 테스트의 단언을 약화.

### R-003 — BullMQ `UnrecoverableError` API 의존 (★★☆)
- **대상 마일스톤:** M4 (분류 → 즉시 격리).
- **현상:** BullMQ 버전에 따라 즉시 실패 처리 API가 다름(`UnrecoverableError`, `MoveToFailedFromUnknownError`, 핸들러에서 옵션 객체 사용 등).
- **대응:**
  - M4 진입 전에 사용 중인 BullMQ 버전의 즉시 격리 메커니즘을 1회 확인.
  - `core/worker.ts`에서 BullMQ가 제공하는 표준 메커니즘만 사용. 자체 hack 금지.
- **롤백 트리거:** BullMQ가 제공하는 어떤 표준 메커니즘도 "재시도 0회"를 달성하지 못한다고 판단되면 발생.
- **롤백 동작:** `attempts: 1`로 동적 옵션 변경(`job.opts.attempts` 조작). 마지막 수단은 PRD `02` §F2.2/§F2.4 정책 일부 약화 + 사람 결정.

### R-004 — Stalled-job 회수 임계 (★★☆)
- **대상 마일스톤:** M6 (`IT-S6`).
- **현상:** `stalledInterval`을 너무 짧게(예: 100ms) 잡으면 BullMQ 내부 락 갱신 주기와 충돌해 false-positive stalled가 발생할 수 있다. 너무 길면 테스트 시간 초과.
- **대응:** Q-STALL-1 (a) 권장 — 테스트는 `STALLED_INTERVAL_MS=500` 권장. 그 이하로 줄이지 않는다.
- **롤백 트리거:** `IT-S6`가 flaky(5회 중 1회 실패)일 때.
- **롤백 동작:** `STALLED_INTERVAL_MS`를 1000ms로 증가, 테스트 polling 타임아웃을 10초까지 늘림. 그래도 flaky면 사람 결정.

### R-005 — 자식 프로세스 SIGTERM 테스트의 환경 의존 (★★☆)
- **대상 마일스톤:** M7 (`IT-S7`).
- **현상:** 자식 프로세스 spawn 시 `tsx`/`ts-node` 등 TypeScript 직접 실행 도구가 SIGTERM 전달을 가로채거나 지연시키는 사례 있음.
- **대응:**
  - `tsx`를 사용한다면 `tsx --no-warnings` 등 옵션 확인.
  - 또는 테스트 픽스처가 `node --import 'tsx/esm' server.ts`로 직접 실행하여 wrapper 효과 최소화.
  - macOS/Linux의 시그널 전달 차이도 확인(CI는 Linux).
- **롤백 트리거:** 자식 프로세스가 SIGTERM에 응답하지 않거나 응답이 일관되지 않을 때.
- **롤백 동작:** Q-OPS-2 (a) 동일 프로세스 시뮬레이션으로 폴백. 단, "진짜 시그널 검증" 강도가 약해짐(불변식 I2.6의 충족도 명시적으로 기록).

### R-006 — 도메인 경계 누출 (★☆☆ 그러나 점진적 위험)
- **대상 마일스톤:** 전 마일스톤.
- **현상:** AI가 생성한 코드가 무심코 `core`에 `WebhookDeliveryError` 같은 도메인 식별자를 넣음.
- **대응:** `IT-R1-domain-boundary`가 즉시 실패. 모든 PR에서 게이트.
- **롤백 트리거:** `IT-R1` 빨강.
- **롤백 동작:** 해당 식별자를 `demo/` 측으로 이동. 인터페이스가 도메인 타입을 노출하면 제네릭화(`<TData>`).

### R-007 — 단일 프로세스 API+워커가 셧다운 시 양쪽 작용 (★☆☆)
- **대상 마일스톤:** M7.
- **현상:** M2~M6 동안 API와 워커가 단일 프로세스(`server.ts`)에서 함께 떠 있어, SIGTERM 시 Fastify와 Worker 양쪽을 모두 닫아야 한다. 순서 잘못 잡으면 in-flight 요청이 짤린다.
- **대응:** `core/shutdown.ts`의 시퀀스를 명시:
  1. HTTP draining 토글.
  2. Worker.pause/close.
  3. 진행 중 작업 대기.
  4. HTTP server close.
  5. Redis quit.
- **롤백 트리거:** `IT-S7`에서 in-flight 요청 짤림 단언이 발생.
- **롤백 동작:** API와 Worker를 별도 프로세스로 분리(`compose.yml`에 `worker` 서비스 추가). PRD `01` MVP 범위에서는 단일 프로세스가 충분하지만, M7에서 필요하면 분리.

### R-008 — `http` 토큰 검사가 `URL`/`createServer` 등과 충돌 (★☆☆)
- **대상 마일스톤:** M1 (`IT-R1` 도입).
- **현상:** `core`가 `http` 모듈 또는 `URL`(Node 표준)을 사용해야 하는 경우, 단순 grep이 false-positive.
- **대응:** `09-cross-cutting.md` §6에 명시 — `BANNED_TOKENS = ['webhook', 'fastify', '_demo', 'Payload']`로 시작. `http` 포함 여부는 사람 결정 후 추가. 본 PLAN은 PRD `04` AC4.1에 따라 `http` 포함을 권장하지만, 충돌 발생 시 사람 결정에 따라 제외 가능.
- **롤백 트리거:** `core`가 정당하게 `URL`/`http` 모듈을 써야 하는 정확한 사례 발견.
- **롤백 동작:** `IT-R1`의 토큰 목록 조정 + PRD에 사유 보고(PRD 수정은 별도 PR).

### R-009 — 새 의존성 추가 압박 (★★☆)
- **대상 마일스톤:** M2~M7.
- **현상:** "이 라이브러리만 추가하면 쉬울 텐데"의 유혹. CLAUDE.md §2 위반.
- **대응:**
  - 새 의존성은 PR 설명/커밋 메시지에 정당성 명시 의무.
  - PLAN의 어떤 마일스톤도 "추가하면 좋을 의존성"을 제안하지 않는다(있는지 자체 검수).
- **롤백 트리거:** 정당성이 명확하지 않은 신규 의존성 발견.
- **롤백 동작:** 의존성 제거 + 표준 라이브러리로 대체. 불가능하면 사람 결정.

## 3. PLAN 자체의 롤백 시나리오

### S-A — 특정 마일스톤이 ETA를 초과
- 트리거: 1주(40시간) 작업해도 해당 마일스톤 Exit Gate를 통과하지 못함.
- 동작:
  1. 사람에게 보고. PLAN 변경 제안.
  2. **PRD 변경 없이** 마일스톤 분해(예: M4를 M4a "분류만" + M4b "백오프" + M4c "HMAC"로 쪼갬).
  3. 그래도 막히면 PRD 수정 검토(별도 절차).

### S-B — 결정 필요 항목이 PLAN 진행 중에 번복됨
- 트리거: 사람이 잠근 Q의 결정이 바뀜.
- 동작:
  1. 영향 받는 마일스톤 식별(`00-decisions-needed.md` §3 매트릭스).
  2. 해당 마일스톤의 영향 단계만 재작성. 다른 마일스톤은 건드리지 않는다.
  3. 회귀 테스트로 다른 마일스톤이 깨지지 않음을 확인.

### S-C — 새 결정 항목 발생
- 트리거: 구현 중 "이건 임의로 정하면 안 되겠다" 판단.
- 동작:
  1. 즉시 멈추고 `00-decisions-needed.md`에 Q-XXX-N 추가(다음 사용 가능 번호).
  2. 동일 내용을 PRD `07` §2에도 추가(별도 PR로 PRD 갱신).
  3. 사람 결정 후 재개.

### S-D — 7개 시나리오 중 하나의 통합 테스트가 만성적 flaky
- 트리거: 5회 연속 실행 중 1회 이상 실패(PRD `03` §7.3 결정성 위반).
- 동작:
  1. 즉시 해당 시나리오의 단언을 약화 **금지**(CLAUDE.md §7-2 위반).
  2. 위 R-001~R-005에 해당하는 리스크인지 식별.
  3. PLAN의 해당 마일스톤에 정의된 롤백 동작 적용.
  4. 그래도 안 되면 사람 결정.

## 4. PRD/CLAUDE.md 위반 발견 시 절차 (CLAUDE.md §7-5)

- **임의로 고치지 않는다.**
- 다음을 사람에게 보고:
  1. 위반의 정확한 파일/행 위치
  2. 위반된 규칙(CLAUDE.md 또는 PRD의 어느 항목)
  3. 제안 수정안 1~2개(트레이드오프 포함)
- 사람 결정 후 별도 PR로 수정.

## 5. 본 PLAN 자체가 깨지면

본 PLAN(`docs/plan/**`)이 실행 도중 명백한 모순/누락을 드러내면:

- 모순을 발견한 시점의 마일스톤을 중단.
- 본 디렉터리에 `00-decisions-needed.md` 또는 새 PLAN 문서로 모순 기록.
- PLAN 갱신은 별도 PR(코드와 섞지 않음).
- 사람 승인 후 재개.
