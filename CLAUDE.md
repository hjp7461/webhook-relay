# CLAUDE.md

이 문서는 Claude Code(및 모든 기여자)가 이 저장소에서 작업할 때 **반드시 따라야 하는 규칙**의 단일 소스입니다.
새 작업을 시작하기 전에 이 문서를 먼저 읽고, 여기에 정의된 규칙과 충돌하는 코드는 생성하지 않습니다.

---

## 1. 프로젝트 개요

Redis(BullMQ) 기반의 **신뢰성 있는 웹훅 전송 작업 큐**입니다.
외부로 웹훅을 전송하다 실패하면 지수 백오프로 재시도하고, 최대 재시도를 초과하면
DLQ(dead-letter queue)로 격리합니다. 처리 상태는 실시간 대시보드와 메트릭으로 관측합니다.

이 프로젝트의 목적은 "가장 빠른 큐"를 만드는 것이 아니라,
**프로덕션급 작업 큐가 갖춰야 할 보장(전달 보장, 멱등성, 재시도, DLQ, 그레이스풀 셧다운,
관측성)을 검증된 라이브러리 위에서 올바르게 조립하는 판단력**을 보여주는 것입니다.

### 트랙 구분
- **메인 트랙 (현재 범위):** BullMQ 기반 웹훅 전송 데모 서비스 + 대시보드.
- **부록 트랙 (미래 범위, 지금은 구현하지 않음):** Raw Redis Streams(XREADGROUP/XACK/
  XAUTOCLAIM)로 큐 내부 원리를 직접 구현하는 학습용 모듈. `packages/streams-internals/`
  자리만 예약해 두고, 메인 트랙이 안정화되기 전에는 착수하지 않는다.

> Claude Code 규칙: 요청이 명시적으로 부록 트랙을 가리키지 않는 한, 항상 메인 트랙
> 범위 안에서만 작업한다. 범위를 임의로 확장하지 않는다.

## 2. 기술 스택 (고정)

| 항목 | 선택 | 비고 |
|------|------|------|
| 언어 | TypeScript (strict) | `any` 금지, 명시적 반환 타입 |
| 런타임 | Node.js 20+ LTS | |
| 큐 | BullMQ | 직접 큐를 재구현하지 않는다 (부록 트랙 제외) |
| Redis 클라이언트 | ioredis | BullMQ 권장 클라이언트 |
| HTTP 서버 | Fastify | 작업 등록 API + 대시보드 서빙 |
| 검증 | Zod | 모든 외부 입력은 경계에서 파싱 |
| 메트릭 | prom-client | Prometheus 포맷 노출 |
| 테스트 | Vitest + Testcontainers | 통합 테스트는 실제 Redis 컨테이너 사용 |
| 패키지 매니저 | pnpm (workspace) | 모노레포 |
| 컨테이너 | Docker Compose | 전체 스택 로컬 재현 |

> 새 의존성을 추가하기 전에 반드시 정당성을 PR 설명 또는 커밋 메시지에 남긴다.
> 표준 라이브러리나 이미 있는 의존성으로 해결 가능하면 새 패키지를 추가하지 않는다.

## 3. 폴더 구조

```

│   │   │   ├── producer.ts     # 작업 등록 + 멱등성 키
│   │   │   ├── worker.ts       # 워커 + 핸들러 실행
│   │   │   ├── retry.ts        # 백오프 정책 + DLQ 이동
│   │   │   ├── metrics.ts      # prom-client 정의
│   │   │   └── shutdown.ts     # 그레이스풀 셧다운
│   │   └── test/
│   ├── demo/                   # 동작하는 웹훅 전송 서비스
│   │   ├── src/
│   │   │   ├── api/            # Fastify 라우트
│   │   │   ├── handlers/       # webhook-delivery 핸들러
│   │   │   ├── receiver/       # 테스트용 웹훅 수신 엔드포인트(데모)
│   │   │   └── chaos.ts        # 장애 주입 (워커 강제 종료 등)
│   │   └── test/
│   └── streams-internals/      # [부록 트랙 자리, 현재 비어 있음]
├── docker/
│   ├── prometheus.yml
│   └── grafana/                # 대시보드 JSON (코드로 버전 관리)
├── docker-compose.yml
├── CLAUDE.md
└── README.md
```

- `core`는 도메인(웹훅)에 의존하지 않는다. 웹훅 관련 로직은 전부 `demo`에 둔다.
- `core`와 `demo`의 경계를 흐리는 코드는 거부한다.

## 4. 코딩 컨벤션

### TypeScript
- `tsconfig`는 `strict: true`. `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`도 켠다.
- `any` 금지. 불가피하면 `unknown` + 타입 가드.
- 공개 함수는 반환 타입을 명시한다.
- 외부 입력(HTTP body, 환경변수, Redis에서 읽은 payload)은 **경계에서 Zod로 파싱**한 뒤
  내부에서는 타입을 신뢰한다.

### 네이밍
- 파일: `kebab-case.ts`
- 타입/인터페이스/클래스: `PascalCase`
- 함수/변수: `camelCase`
- 상수: `UPPER_SNAKE_CASE`
- 큐/작업 이름 같은 문자열 식별자는 한 곳(`constants.ts`)에 모아 매직 스트링을 피한다.

### 에러 핸들링
- 도메인 에러는 커스텀 에러 클래스로 표현한다 (`WebhookDeliveryError` 등).
- 재시도 가능한 에러와 불가능한 에러(예: 4xx 응답)를 **명시적으로 구분**한다.
  불가능한 에러는 재시도하지 않고 즉시 DLQ 또는 실패 처리한다.
- 절대 에러를 조용히 삼키지 않는다(`catch {}` 금지). 로깅하거나 다시 던진다.

### 로깅
- 구조화 로깅(JSON). `jobId`, `attempt`, `idempotencyKey`를 항상 컨텍스트에 포함한다.
- 민감 정보(웹훅 시크릿, 헤더의 인증 토큰)는 로그에 남기지 않는다.

### 비동기/동시성
- 떠 있는 Promise(floating promise) 금지. 모든 Promise는 await하거나 의도적으로 처리한다.
- 공유 상태는 Redis에 둔다. 워커 프로세스의 인메모리 상태에 의존하지 않는다
  (워커는 언제든 죽고 재시작될 수 있다).

## 5. 테스트 정책

테스트는 이 프로젝트의 핵심 어필 포인트다. "동작한다"가 아니라 "장애 상황에서도
보장이 지켜진다"를 증명해야 한다.

### 분류
- **단위 테스트 (`*.unit.test.ts`):** 순수 로직(백오프 계산, 멱등성 키 생성, 에러 분류).
  Redis 없이 빠르게 실행.
- **통합 테스트 (`*.integration.test.ts`):** Testcontainers로 **실제 Redis**를 띄워
  큐 동작을 검증. 모킹된 Redis는 신뢰하지 않는다.

### 반드시 통합 테스트로 검증할 시나리오
1. 작업 등록 → 워커 처리 → 완료 (happy path)
2. 동일 멱등성 키로 중복 등록 시 한 번만 처리되는가
3. 재시도 가능 에러 → 지수 백오프로 N회 재시도되는가
4. 최대 재시도 초과 → DLQ로 이동하는가
5. 재시도 불가 에러(4xx) → 재시도 없이 즉시 격리되는가
6. 워커가 처리 중 강제 종료 → 작업이 유실되지 않고 다른 워커가 회수하는가
7. SIGTERM 수신 → 진행 중 작업을 마치고 그레이스풀하게 종료하는가

### 규칙
- 새 기능 PR에는 대응하는 테스트가 반드시 포함된다.
- 동시성/장애 복구 로직은 **테스트를 먼저 작성**한다(아래 §7 워크플로우 참조).
- 시간 의존 테스트(백오프 지연 등)는 fake timer로 결정론적으로 만든다.
- 통합 테스트는 서로 격리한다(각 테스트가 고유한 큐 이름/prefix 사용).

## 6. 커밋 / PR 규칙

- Conventional Commits 형식: `feat:`, `fix:`, `test:`, `docs:`, `refactor:`, `chore:`.
- 커밋은 논리적으로 원자적이게. "여러 관심사를 한 커밋에" 금지.
- 각 단계(README 로드맵의 1~4단계)가 끝날 때 데모는 **항상 동작하는 상태**를 유지한다.
- AI에게 위임한 작업이라도, 커밋 메시지는 "무엇을 왜 바꿨는지" 사람이 이해할 수 있게 쓴다.

## 7. AI 협업 워크플로우 (Claude Code 사용 규칙)

이 저장소는 Claude Code와 함께 개발하되, **설계와 검증의 책임은 사람에게 있다**는 원칙으로
운영한다. Claude Code는 다음 규칙을 따른다.

1. **설계는 사람이 먼저 한다.** 동시성·장애 복구처럼 까다로운 로직은 사람이 의도와
   불변식(invariant)을 먼저 정의한다. Claude Code는 그 불변식을 깨는 구현을 제안하지 않는다.
2. **테스트 우선.** 위 §5의 핵심 시나리오는 구현 전에 테스트를 먼저 작성한다.
   구현은 그 테스트를 통과시키는 방향으로만 진행한다.
3. **범위를 넘지 않는다.** 요청받지 않은 기능, 추상화, 의존성을 임의로 추가하지 않는다.
   "있으면 좋을 것 같은" 코드는 만들지 않는다.
4. **불확실하면 멈추고 묻는다.** 규칙이 모호하거나 트레이드오프가 있는 결정은 임의로
   정하지 않고 사람에게 선택지를 제시한다.
5. **이 문서를 위반하는 기존 코드를 발견하면** 조용히 고치지 말고 먼저 보고한다.
6. **존대말을 반드시 사용한다.** 사용자에게 보내는 모든 텍스트 응답(보고, 질문, 제안,
   설명, 거절, 에러 보고 포함)은 한국어 존대말로 작성한다. 반말·평어·명령형 종결
   금지. 코드/커밋 메시지/문서 본문은 본 규칙의 대상이 아니며, 본 저장소의 기존
   문체(`-한다`/`-이다` 평서체)를 유지한다.

> 이 워크플로우 자체가 포트폴리오의 일부다. "AI가 코드를 짜줬다"가 아니라
> "AI를 규율 있는 협업자로 다뤘다"를 보여주는 것이 목표다.

## 8. 보안 / 운영 주의

- 웹훅 전송 시 페이로드 서명(HMAC) 헤더를 붙여 수신자가 위변조를 검증할 수 있게 한다.
- 외부로 나가는 요청에는 타임아웃을 반드시 건다(무한 대기 금지).
- 환경변수는 `.env.example`로 문서화하고, 실제 시크릿은 커밋하지 않는다.
- Redis 연결 실패 시 무한 재연결로 폭주하지 않도록 백오프를 둔다.
